# 🧩 Шардирование БД — когда одна база уже не вывозит

**Шардирование (sharding)** — горизонтальное разбиение данных на несколько независимых частей (**шардов**), которые живут на разных базах/узлах. Когда вертикальное масштабирование (мощнее сервер, больше RAM) и read-реплики уже не спасают, данные **распределяют**: например, пользователи с одними `user_id` — в одном шарде, с другими — в другом. Плата за это — резко усложнившаяся работа с данными и инфраструктурой.

> [!info] Не путать три способа масштабирования — это разные вещи
> | Приём | Что делает | Решает |
> | :--- | :--- | :--- |
> | **Репликация** | копии **тех же** данных на другие узлы | масштабирование **чтения**, отказоустойчивость |
> | **Партиционирование** | режет **одну таблицу** на части **внутри одной БД** (Postgres `PARTITION BY`) | размер таблицы/индекса, скорость запросов по ключу |
> | **Шардирование** | разносит данные по **разным БД/серверам** | масштабирование **записи** и объёма сверх одного сервера |
>
> Пост это местами смешивает: партиционирование — про одну базу, шардирование — про **несколько независимых баз**. Часто их комбинируют (шард → внутри партиции → внутри реплики).

## 🔑 Главный вопрос: по какому ключу шардировать (shard key)

`shardKey` определяет, в какой шард попадёт строка. От его выбора зависит **всё** — плохой ключ убивает производительность двумя способами: заставляет ходить во все шарды и создаёт перекос нагрузки.

### Стратегии выбора шарда (в посте только hash/modulo — а их больше)

| Стратегия | Как | Плюсы | Минусы |
| :--- | :--- | :--- | :--- |
| **Hash / modulo** — `user_id % N` | остаток от деления | равномерно, просто | ⚠️ **при добавлении шарда** (`% 4` → `% 5`) меняется формула → **переезжают почти все данные** |
| **Consistent hashing** | кольцо хешей | добавление шарда двигает **малую долю** ключей | сложнее реализовать |
| **Range (диапазоны)** | `id 0–1M → shard_0` | удобны range-запросы, легко добавлять шард | легко получить **hot shard** (свежие id — все в последний) |
| **Directory / lookup** | таблица-справочник «ключ → шард» | максимально гибко, легко переносить | лишний lookup, справочник — новая точка отказа |
| **Geo / tenant** | по региону / клиенту | data locality, изоляция | перекос по крупным клиентам |

> [!caution] `user_id % N` из поста — самый коварный на будущее
> Модуло-схема отлично раскладывает данные **сегодня**, но при росте кластера (`% 4` → `% 8`) ключ у большинства строк меняется → **массовый респлит с переносом почти всего датасета**. Именно поэтому в проде чаще берут **consistent hashing** или **directory-based**, либо шардируют на **много «виртуальных» шардов заранее** (например, 1024 логических бакета на 4 физических узла — растёшь, перемещая бакеты, а не пересчитывая всё).

> [!warning] Hot shard — перекос нагрузки
> Если ключ распределяет неравномерно, один узел «горит» (много данных и запросов), остальные простаивают — это **hot shard**, бутылочное горлышко. Классика — **celebrity problem**: шардируем посты по `author_id`, а у блогера-миллионника один шард ловит весь трафик. Лечится выбором **более гранулярного/высококардинального** ключа и продумыванием на этапе проектирования — перераспределять данные потом **дорого**.

## 🔍 Боль #1: запросы не по shard key (scatter-gather)

Запрос по ключу шардирования — прямой, идём в один шард:

```sql
-- shard key = user_id → сразу знаем куда идти
SELECT * FROM orders WHERE user_id = 123;
```

Запрос **не по ключу** — не знаем, где лежат строки, значит идём **во все шарды**, выполняем, собираем и склеиваем результат (**scatter-gather**):

```sql
-- status не ключ шардирования → +N сетевых вызовов
SELECT * FROM orders WHERE status = 'FAILED';
```

- 10 шардов = **+10 сетевых вызовов**, латентность по **самому медленному**, и риск, что какой-то отвалится.
- Изредка — ОК. Если таких запросов много — **пересматривай схему хранения** (денормализация, вторая копия данных под другой ключ, поисковый индекс типа Elastic/OpenSearch рядом).

## 🔗 Боль #2: JOIN между шардами

`user` и его `orders` на **разных** шардах → простым `JOIN` уже не соединить. Решения:

- **Co-location (совместное размещение)** — шардировать связанные таблицы **одним ключом**: у `orders` shard key должен быть **`user_id`**, а не `order_id`, тогда пользователь и его заказы лежат в одном шарде и `JOIN` локальный. (В Citus — `distribution column`; в посте эта мысль верная и ключевая.)
- **Денормализация** — дублировать нужные поля, чтобы не джойнить.
- **Reference-таблицы** — маленькие справочники копировать на все шарды целиком.
- **JOIN на стороне приложения** — достать по частям и склеить в коде (медленно, но иногда проще).

> [!note] Что ещё «ломается» после шардирования
> - **Транзакции между шардами** — нужен распределённый коммит (**2PC**), это медленно и хрупко; по возможности держи транзакцию в пределах одного шарда.
> - **Уникальность / автоинкремент** — глобальный `AUTO_INCREMENT` не работает; берут **UUID**, Snowflake-id, или отдельный генератор.
> - **Вторичные индексы** — глобального индекса нет; поиск не по ключу = scatter-gather или отдельный индекс-сервис.
> - **`ORDER BY` / `LIMIT` / агрегаты** через все шарды — считаются частично на каждом и мержатся координатором.

## ⚙️ Где это «включить»: SQL БД сама не шардирует

В PostgreSQL (и SQL-СУБД вообще) шардирования **из коробки нет**. Два основных пути (как в посте) плюс третий современный:

### 🟣 1. Свой routing layer в приложении

Код обращается как обычно — `userRepo.save(user)` — а под капотом библиотека считает ключ и идёт в нужный шард:

```java
int shardId = userId % shardCount;
DataSource dataSource = shardRouter.getDataSource(shardId);
```

Гибко и «бесплатно» по лицензии, но **всю логику** (роутинг, scatter-gather, респлит, failover) пишешь и поддерживаешь **сам** — обычно это отдельная платформенная библиотека.

### 🟣 2. Расширение Citus поверх Postgres

[**Citus**](https://github.com/citusdata/citus) (open-source, Microsoft) превращает Postgres в распределённую БД: обращаешься к **узлу-координатору**, он сам роутит и распараллеливает по воркерам.

```sql
CREATE EXTENSION citus;
SELECT create_distributed_table('orders', 'user_id');  -- distribution column
```

Не «галочка»: нужен кластер (координатор + воркеры), понимание co-location и типов таблиц (distributed / reference / local). Managed-вариант — Azure Cosmos DB for PostgreSQL.

### 🟣 3. Distributed SQL — шардинг «встроен» в саму БД

Если проект новый — можно не прикручивать шардинг к классической БД, а взять СУБД, где он **нативный и прозрачный**:

- **[CockroachDB](https://github.com/cockroachdb/cockroach)**, **[YugabyteDB](https://github.com/yugabyte/yugabyte-db)** — Postgres-совместимые, авто-шардинг (range) + консенсус (Raft), респлит без ручной боли.
- **[Vitess](https://github.com/vitessio/vitess)** — горизонтальное масштабирование MySQL (на нём росли YouTube/Slack).
- **MongoDB / Cassandra / ClickHouse** — шардирование как штатная возможность.

## 🖥️ Как попробовать на своих системах

Шардированный кластер имеет смысл поднимать на десктопе/сервере; на роутере — только «пощупать» одиночный Postgres.

| Система | Как поднять для экспериментов |
| :--- | :--- |
| **Gentoo** (основная) | `emerge dev-db/postgresql`; **Citus** — из исходников/контейнера (в Portage нет), проще через **Docker**: `docker run citusdata/citus`. Для distributed SQL — контейнеры CockroachDB/YugabyteDB |
| **Debian / Ubuntu** | у **Citus** есть свой apt-репозиторий (`install.citusdata.com`) — самый гладкий путь потренироваться; либо Docker-compose с координатором+воркерами |
| **Arch** (план с июня 2026) | `pacman -S postgresql`; Citus — AUR/Docker; CockroachDB — бинарь/контейнер |
| **Entware / RT-AX56U** | ⚠️ Не для этого: `opkg` есть лёгкий `postgresql`, но 512 МБ RAM и слабый armv7 не тянут кластер. Максимум — **одиночный** Postgres как учебный стенд; шардинг/Citus — нет |

## 💡 Практический вывод

- Шардируй, **когда действительно упёрся** (объём/запись сверх одного сервера), а не «на всякий». Сначала — индексы, вертикаль, реплики чтения, партиционирование, кэш.
- **Выбор shard key — решение №1**: под свои главные запросы (co-location для JOIN), высокая кардинальность против hot shard, запас на рост (виртуальные шарды / consistent hashing).
- Готовься платить сложностью: cross-shard запросы, распределённые транзакции, респлит, мониторинг N узлов.
