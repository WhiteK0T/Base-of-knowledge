# 🐘 VACUUM FULL — возврат места на диске в PostgreSQL (bloat, dead tuples, pg_repack)

**Симптом:** таблица занимает на диске десятки гигабайт, хотя реальных данных там пара гигабайт — особенно после массового `DELETE`/`UPDATE`. Это **раздувание (bloat)**. Разберём, почему так происходит и как вернуть место.

## 🧩 Почему место не освобождается само

PostgreSQL использует **MVCC**: при `DELETE` строка не стирается физически, а помечается как **«мёртвая» (dead tuple)**; при `UPDATE` создаётся новая версия строки, а старая остаётся мёртвой. Старые версии нужны, пока их видят параллельные транзакции. В итоге файл таблицы продолжает занимать место, даже если «живых» строк мало.

- **`autovacuum`** (фоновый, включён по умолчанию) и обычный **`VACUUM`** вычищают мёртвые строки и **помечают место как свободное для повторного использования** — но **не отдают его обратно ОС** (файл не уменьшается; место переиспользуется будущими вставками в эту же таблицу).
- Чтобы **реально вернуть место на диск**, файл нужно **переписать** — это и делает `VACUUM FULL`.

> [!note] Обычный VACUUM ≠ освобождение диска
> `VACUUM` (без `FULL`) **не блокирует** таблицу и нужен для регулярного обслуживания — но размер файла он почти не уменьшает (отдаёт ОС только полностью пустые «хвостовые» страницы). Раздувание он не лечит, лишь не даёт расти дальше.

## 🔍 Диагностика

```sql
-- размер таблицы с индексами и без
SELECT pg_size_pretty(pg_total_relation_size('my_table')) AS total,
       pg_size_pretty(pg_relation_size('my_table'))       AS table_only;

-- сколько живых/мёртвых строк и когда последний (auto)vacuum
SELECT relname, n_live_tup, n_dead_tup, last_vacuum, last_autovacuum
FROM pg_stat_user_tables
WHERE relname = 'my_table';
```

> [!tip] Точная оценка bloat — расширение `pgstattuple`
> `CREATE EXTENSION pgstattuple;` затем `SELECT * FROM pgstattuple('my_table');` — покажет реальный процент «мусора» (`free_percent`, `dead_tuple_percent`). Так понятно, **стоит ли вообще** затевать тяжёлый `VACUUM FULL`.

## 🛠️ Как вернуть место

```sql
VACUUM VERBOSE my_table;   -- обычная чистка (НЕ блокирует), покажет статистику
VACUUM FULL my_table;      -- переписывает таблицу, ВОЗВРАЩАЕТ место ОС
ANALYZE my_table;          -- обновить статистику планировщика после переписи
```

> [!warning] `VACUUM FULL` — это «стоп-таблица»
> Он берёт **`ACCESS EXCLUSIVE` lock**: на время операции таблица **полностью недоступна — даже на чтение**. Плюс он строит **новую копию таблицы и индексов рядом**, поэтому нужно **свободное место на диске ≈ размер самой таблицы**. На большой «горячей» таблице в проде это означает простой. Запускать — только в **окно обслуживания**.

## 🚀 Прод-вариант: `pg_repack` (онлайн, без долгой блокировки)

[**pg_repack**](https://github.com/reorg/pg_repack) — расширение + CLI-утилита, которая делает то же, что `VACUUM FULL` (переписывает таблицу/индексы, отдаёт место ОС), но **онлайн**: таблица остаётся доступной для чтения и записи. Эксклюзивную блокировку берёт лишь на **доли секунды в начале и в конце**.

```bash
# создать расширение в БД (один раз)
psql -d mydb -c "CREATE EXTENSION pg_repack;"

# переупаковать одну таблицу (online)
pg_repack -d mydb -t my_table

# или всю БД
pg_repack -d mydb
```

> [!caution] Условия pg_repack
> - Таблице нужен **PRIMARY KEY** или не-частичный **UNIQUE**-индекс.
> - Нужно **примерно вдвое больше** свободного места (работает через временную копию).
> - Версия расширения и CLI должны совпадать; пишет в БД, так что лучше прогнать сперва на стейдже.

## 📦 Установка pg_repack по системам

`VACUUM`/`VACUUM FULL` встроены в PostgreSQL — ставить ничего не надо. Отдельно ставится только **pg_repack**:

| Система | Установка pg_repack |
| :--- | :--- |
| **Gentoo** (основная) | `emerge dev-db/pg_repack` (USE-флаг под нужный слот `dev-db/postgresql`) |
| **Debian / Ubuntu** | `sudo apt install postgresql-NN-repack` (где `NN` — мажор версии, напр. `postgresql-17-repack`); из PGDG-репозитория |
| **Arch** (план с июня 2026) | AUR: `yay -S pg_repack` (или `pg_repack-git`) |
| **Entware / RT-AX56U** | **N/A** — в opkg-репозитории есть сам `postgresql`, но расширения `pg_repack` нет. На роутере при раздувании — обычный `VACUUM FULL` в окно обслуживания (учти 256 МБ flash: места под копию может не хватить → лучше дамп/рестор) |

## 🧯 Профилактика (чтобы не раздувалось)

- **Партиционирование по времени** для логов/метрик: вместо массового `DELETE` старых данных — `DROP`/`DETACH` целой партиции (мгновенно, без bloat).
- **`TRUNCATE`** вместо `DELETE`, если удаляешь **всю** таблицу — место отдаётся сразу, мёртвых строк не остаётся.
- Настроить **autovacuum агрессивнее** на таблицах с интенсивным `UPDATE`/`DELETE` (`autovacuum_vacuum_scale_factor`, `..._cost_limit`).
- Альтернативы переписи: **`CLUSTER`** (тоже блокирует, но заодно упорядочивает по индексу), **pg_squeeze** (онлайн, как pg_repack).

## 💡 Кратко

- Место «съедают» **мёртвые версии строк** (MVCC) — обычный `VACUUM`/autovacuum их чистит, но **не уменьшает файл**.
- **Вернуть диск ОС:** `VACUUM FULL` — просто, но **блокирует таблицу** и требует места под копию.
- **В проде без простоя:** **pg_repack** (онлайн) — нужен PRIMARY KEY и ~2× места.
- **Лучше предотвратить:** партиции + `DROP`, `TRUNCATE`, грамотный autovacuum.
