# ☁️ Unlim Cloud — «безлимитное» облако из Telegram (как работает, риски, аналоги)

Из Telegram-канала «Не баг, а фича» разошёлся пост про десктопное приложение **Unlim Cloud** ([inulute/unlim-cloud](https://github.com/inulute/unlim-cloud), MIT, ~950★), которое якобы превращает мессенджер в **бесплатное безлимитное облачное хранилище**: логинишься своим Telegram, льёшь файлы в «Сохранённые сообщения», а Telegram почти не ограничивает объём — получается «бесконечный диск».

Идея не новая и рабочая, но по факту **пост сильно упрощает**, а главное — **умалчивает про модель доверия**. Ниже — что это на самом деле, где подвох и чем заменить.

> [!warning] Коротко
> «Открытый десктоп-клиент» из репозитория — это **пустышка-обёртка**: она лишь открывает сторонний сайт `unlim-cloud.web.app`. Вся работа с Telegram (вход по номеру телефона, код, 2FA, заливка файлов) идёт **в коде, который отдаёт чужой сервер** и которого в открытом репозитории **нет**. Вы доверяете **полную сессию** своего аккаунта третьей стороне.

## 🔬 Что это на самом деле (разбор кода)

Заглянул в исходники — архитектура оказалась совсем не такой, как звучит в посте:

1. **Десктоп-приложение = Tauri-обёртка вокруг вебвью и больше ничего.** Весь Rust-бэкенд (`src-tauri/src/main.rs`) делает ровно одно: грузит локальный `index.html`. Никакого кода Telegram, шифрования или хранения в нативной части **нет**. В `tauri.conf.json` даже все разрешения (fs, http, shell, dialog) выключены — приложению нечего и незачем делать локально.
2. **`index.html` после проверки версии просто редиректит на сайт** `https://unlim-cloud.web.app`. То есть «приложение» — это, по сути, **ярлык-браузер** на чужой веб-сервис. Сам разработчик в README честно пишет: *«This application is not affiliated with the official Unlim Cloud»* — десктоп-обёртка **неофициальная**, а «official Unlim Cloud» — это тот самый веб-сайт.
3. **Веб-сайт `unlim-cloud.web.app`** (SPA на Firebase Hosting) — вот где вся логика. В его JS видно, что он использует **TDLib** (официальная библиотека Telegram, WASM-сборка `tdweb`) через обёртку **Airgram** с захардкоженными `api_id`/`api_hash`. Это значит:
   - вход — **по номеру телефона + код из Telegram (+ пароль 2FA, если включён)**, как в настоящем клиенте;
   - создаётся **полноценная пользовательская сессия** (не бот!), которая хранится в браузере (IndexedDB);
   - файлы заливаются **напрямую в ваш Telegram** (клиент → серверы Telegram); отдельного «облака разработчика», куда сливались бы файлы, в текущем коде **не видно** — своих бэкенд-адресов для файлов в JS нет.

> [!note] Почему это важно
> «Логин по Telegram ID» из поста звучит безобидно, будто отдаёшь только имя/номер. На деле это **полный вход в аккаунт по номеру + коду**, дающий доступ **ко всей переписке и контактам**, а не только к хранилищу. Это принципиально не то же самое, что «Войти через Telegram» (Login Widget), который отдаёт лишь ID и имя.

## ♾️ «Безлимит» — правда или маркетинг?

Частично правда, но с оговорками:

- **Общего лимита на объём** «Сохранённых сообщений» Telegram официально не публикует — в этом смысле хранилище действительно «условно бесконечное».
- Но есть **лимит на размер одного файла**: **2 ГБ** для обычного аккаунта и **4 ГБ** для Telegram Premium. Большие файлы такие сервисы **режут на части** (чанки) — значит, при скачивании их надо собирать обратно тем же сервисом (vendor lock-in на его формат).
- Действует **fair-use**: использование Telegram как файлового хостинга/CDN противоречит духу разумного пользования; при массовых заливках аккаунт может получить **флуд-лимиты** или ограничения. Для личного бэкапа это обычно некритично, но «бесконечный торрент-сид на халяву» — путь к бану.

## ⚠️ Риски безопасности (главное)

> [!caution] Вы отдаёте полную сессию аккаунта коду с чужого сервера
> Даже если **сейчас** JS честно держит сессию в браузере и не сливает файлы на сторону — код отдаёт оператор сайта `unlim-cloud.web.app`, и он может **в любой момент подменить его** на версию, которая выгрузит вашу **сессию, пароль 2FA и файлы**. Это разовый **полный угон аккаунта** (все чаты, контакты, привязанные сервисы). «Клиентская сессия» защищает лишь до тех пор, пока владелец кода добросовестен и не взломан.

Остальные подводные камни:

- **«Open source» вводит в заблуждение.** Открыт только пустой десктоп-враппер. **Та часть, что реально держит ваши данные** (веб-приложение), в репозиторий не выложена — её нельзя проверить или самому захостить. Аудировать нечего.
- **Десктоп-версия не даёт ничего сверх браузера** — это тот же сайт в вебвью. Плюсом идёт **устаревшая** обёртка: релиз `v2.0.0` от **марта 2024**, только под **Windows** (Linux/macOS «coming soon» с тех пор так и нет).
- **Доступ шире, чем нужно для хранилища.** TDLib-сессия видит **все ваши сообщения** — вредоносное обновление могло бы читать переписку, а не только «диск».
- **Приватность.** Файлы лежат в вашем Telegram незашифрованными для самого Telegram (это не E2E — «Сохранённые сообщения» не секретный чат). Плюс метаданные о заливках.

## ✅ Как сделать безопаснее / чем заменить

> [!tip] Если очень хочется попробовать
> Логиньтесь **отдельным «расходным» аккаунтом** (не основным), с **включённым 2FA**, и относитесь к нему как к одноразовому. Никогда не вводите номер/код основного аккаунта в сторонние веб-клиенты.

Более честные альтернативы «Telegram как диск»:

- **Просто «Сохранённые сообщения» самого Telegram.** Тот же «безлимит» без посредника: официальный клиент, ваша сессия, никакого чужого JS. Минус — нет папок/галереи «как в облаке», но для бэкапа файлов достаточно.
- **Инструменты на боте (bot token), а не на вашей сессии.** Бот не имеет доступа к вашей переписке, а токен можно отозвать — компромисс сессии исключён. Примеры: self-hosted **TeleDrive**, **telegram-drive** и подобные. Расплата — **лимиты Bot API на размер файла** (~20 МБ на скачивание / ~50 МБ на отправку без собственного Bot API-сервера), поэтому под большие файлы нужен свой локальный Bot API-сервер.
- **Нормальное облако/бэкап-стек.** Если цель — надёжное хранение, а не «халявный безлимит»: **[rclone](https://rclone.org/)** (десятки бэкендов + `crypt` для клиентского шифрования) поверх любого S3/облака. Есть в Portage (`net-misc/rclone`), в Arch (`pacman -S rclone`), в Debian/Ubuntu (`apt install rclone`), и даже в **Entware** (`opkg install rclone`) — так что бэкап можно гонять прямо с роутера **RT-AX56U** на USB-диск/в облако.

## 🖥️ По платформам (для сведения)

| Система | Как запустить Unlim Cloud |
| :--- | :--- |
| **Windows** | Единственный официальный десктоп-билд (Tauri, `v2.0.0`) |
| **Gentoo / Arch / Debian-Ubuntu** | Отдельного клиента нет → просто откройте `unlim-cloud.web.app` в браузере (десктоп-обёртка — это тот же сайт) |
| **Entware / RT-AX56U** | **N/A** — нет браузера/десктопа; для «Telegram как хранилище» на роутере смотрите бот-инструменты или `rclone` |
