# 🔐 LUKSbox — кроссплатформенный шифроконтейнер для недоверенного хранилища

**LUKSbox** ([github.com/PentHertz/LUKSbox](https://github.com/PentHertz/LUKSbox)) — инструмент на **Rust** для **зашифрованных контейнеров (vaults)**: хранить чувствительные файлы **в облаке или на общих/чужих носителях, не доверяя хосту**. Файлы шифруются **до** выгрузки, под **твоими** ключами; провайдер видит один непрозрачный blob и не расшифрует его даже по запросу властей. Монтируется как **реальный диск** на Linux, macOS, Windows. Автор — **PentHertz** (известная ИБ-компания). Apache-2.0, ~690★, активный. **Версия v0.4.0 (pre-1.0).**

> [!warning] Название вводит в заблуждение: это НЕ формат LUKS
> Несмотря на «LUKS» в имени, LUKSbox **не использует Linux-формат LUKS/LUKS2** и несовместим с `cryptsetup`. Это **собственный** контейнерный формат: vault — это файл **`.lbx`**, опционально с отдельным заголовком **`.hdr`**, постквантовым сайдкаром **`.kyber`** и anchor-сайдкаром против отката. Имя — маркетинговая отсылка к «шифрованному тому», а не совместимость с LUKS. Если ждёшь LUKS2/`cryptsetup` — это другое.

> [!caution] Главный нюанс: pre-1.0 и ВНЕШНЕГО аудита ещё НЕ было
> Для крипто-инструмента это критично, и авторы честны в README:
> - Статус **pre-1.0**, on-disk формат «заморожен», примитивы — стандарты NIST/RFC на RustCrypto, **9 внутренних раундов аудита**, 30M+ фаз-итераций, 200+ тестов. Гигиена хорошая.
> - **НО**: «Third-party audit: **not yet performed**» — независимого платного аудита пока нет. Золотое правило: **не доверяй молодой непроверенной криптографии единственную копию важных секретов.**
> - Сам README предупреждает: контейнер — **единая точка отказа**; если `.lbx` побьётся или все keyslot'ы станут недоступны, данные пропали. **Всегда держи незашифрованную копию** того, что нельзя потерять, в доверенном месте. Vault — это «путешествующая», а не мастер-копия.

## 🧩 Как открывается vault (keyslots)

Keyslot'ы не шифруют файлы напрямую — они **оборачивают мастер-ключ (MVK)**; из MVK через HKDF-SHA256 выводятся все подключи. Разблокировать MVK можно (в т.ч. комбинируя):

| Метод | Что это |
| :--- | :--- |
| **Пароль / PIN** | Argon2id (по умолч. 256 MiB / 3 / 4) |
| **FIDO2 hmac-secret** | аппаратный ключ: **YubiKey, Titan, Nitrokey, Windows Hello** (CTAP2) |
| **TPM 2.0 / SEP** | привязка к чипу устройства (Linux TPM; опц. PIN; режим TPM+FIDO2) |
| **Гибридный постквант** | **ML-KEM-768 / 1024 (FIPS 203)** — PQ-половина смешивается с классикой через HKDF |

Отзыв слота (`revoke`) необратимо лишает этот материал доступа к MVK.

## 🛡️ Что реально даёт (по threat-model)

- **Провайдер/облако не читает файлы** даже под повесткой — шифрование на твоей стороне.
- **AEAD на каждый чанк** (AES-256-GCM-SIV по умолч., либо AES-256-GCM / ChaCha20-Poly1305) + AAD `file_id‖chunk_index‖generation` → ловит подмену/перестановку/replay чанков.
- **Tamper-evident**: HMAC-SHA256 над заголовком; **anchor-сайдкар** детектит откат всего vault.
- **Detached-header режим**: сам `.lbx` — «неотличим от случайного», без magic-байтов и версий.
- **Постквант «harvest now, decrypt later»**: гибридный слот не хуже классики, даже если PQ сломают.
- Защита ключа в памяти: `memfd_secret(2)` на Linux 5.14+ (исключает MVK из coredump/hibernate), `Zeroize` секретов.

> [!info] Ниша среди аналогов
> Похоже на **Cryptomator / gocryptfs** (заточка под облако) и **VeraCrypt** (единый файл-контейнер), но с упором на **аппаратные ключи (FIDO2/TPM)**, **постквант** и tamper/rollback-детект. Полное сравнение (vs LUKS2 / VeraCrypt / age / gocryptfs / Cryptomator / BitLocker / FileVault) — на [luksbox.penthertz.com/compare](https://luksbox.penthertz.com/compare/). Для «просто пошифровать папку локально» проще age/gocryptfs; LUKSbox — когда важны недоверенное хранилище + железные ключи + запас на постквант.

## 🖥️ Применимость на системах владельца

| Система | Как использовать |
| :--- | :--- |
| **Gentoo (основная) / Debian-Ubuntu / Arch** | ✅ **основная цель**: сборка через `cargo` (Rust), монтирование через FUSE, полноценный **TPM 2.0** и **FIDO2 (YubiKey/Nitrokey)** на Linux. Идеально ложится на твой профиль (шифрование, недоверие к облаку) |
| **Windows** | ✅ монтирование через **WinFsp** (надо поставить), Windows Hello как FIDO2. Мелочь: 1 unmaintained транзитивная зависимость (`registry` через WinFsp) |
| **Entware / RT-AX56U** | ➖ собрать Rust-тулзу на armv7-роутере и монтировать FUSE непрактично. **НО** роутер отлично подходит как **недоверенное хранилище**: `.lbx`-blob можно держать на его USB-диске (это и есть сценарий «чужой носитель»), а монтировать/расшифровывать на десктопе |

> Ты уже используешь [git-crypt](../VCS/Git/git-crypt%20%E2%80%94%20%D0%BF%D1%80%D0%BE%D0%B7%D1%80%D0%B0%D1%87%D0%BD%D0%BE%D0%B5%20%D1%88%D0%B8%D1%84%D1%80%D0%BE%D0%B2%D0%B0%D0%BD%D0%B8%D0%B5%20%D1%84%D0%B0%D0%B9%D0%BB%D0%BE%D0%B2%20%D0%B2%20%D1%80%D0%B5%D0%BF%D0%BE%D0%B7%D0%B8%D1%82%D0%BE%D1%80%D0%B8%D0%B8.md) для секретов в репо. LUKSbox — про другое: **бэкап секретов в облако/на USB** одним контейнером с аппаратным ключом. Пока pre-1.0 — гоняй на **некритичных** данных или как второй слой поверх уже проверенного (age/VeraCrypt), а единственную копию важного ему не доверяй.
