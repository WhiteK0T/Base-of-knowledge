# 🔎 lsof — кто, что и куда открыл (шпаргалка)

**lsof** (*list open files*) показывает **все открытые файлы** в системе и кто их держит. В Unix «файл» — это почти всё: обычные файлы, каталоги, **сокеты** (сеть), пайпы, устройства. Поэтому одной утилитой видно и какой процесс слушает порт, и кто блокирует размонтирование диска, и какие удалённые файлы всё ещё «висят» и едят место. Незаменим в диагностике и при разборе **подозрений на взлом**.

> [!warning] Запускай от root — иначе видно только своё
> Без `sudo`/root lsof покажет **только твои** процессы. Полная картина системы — `sudo lsof ...`. Многие примеры ниже без root отдадут лишь часть строк (и кучу `permission denied`).

## 🚀 База (из шпаргалки — проверено и уточнено)

| Задача | Команда |
| :--- | :--- |
| Файлы, открытые процессом | `lsof -p <PID>` |
| Кто на порту 80 | `lsof -i :80` |
| Все сетевые соединения (быстро) | `lsof -i -P -n` |
| Кто держит файл | `lsof /path/to/file` |
| Все файлы пользователя | `lsof -u <username>` |
| Удалённые, но открытые файлы (пожиратели места) | `lsof +L1` |

> [!info] `-P` и `-n` — почему почти всегда стоит добавлять
> `-n` — **не резолвить** IP в имена (без DNS), `-P` — **не резолвить** номера портов в имена (`443` вместо `https`). Без них lsof лезет в DNS и `/etc/services` и **сильно тормозит**, особенно `lsof | grep ...`.

## 🧪 Факты против хайпа — где шпаргалка вводит в заблуждение

> [!caution] «`lsof /path` найдёт файл, даже если он удалён» — нет
> Если файл **уже удалён**, у него **нет пути** — обращаться по `lsof /path/to/file` не к чему. Удалённые-но-открытые файлы ищут **иначе**:
> - `lsof +L1` — все файлы с числом ссылок < 1 (т.е. удалённые, но ещё открытые);
> - `lsof -p <PID>` — в выводе у таких записей стоит пометка **`(deleted)`**.
> `lsof /path` работает, только **пока файл существует** в ФС.

> [!caution] «Найти SSH-ключ в памяти» — формулировка неточная
> `lsof | grep id_rsa` показывает процессы, у которых файл `id_rsa` **открыт как дескриптор** (читается с диска), а **не содержимое памяти**. Это всё равно полезно для аудита («кто вообще трогает мой приватный ключ»), но «ключ в RAM» так не увидишь. И добавь `-n`, иначе долго:
> ```bash
> sudo lsof -n | grep -i 'id_rsa\|id_ed25519\|\.ssh/'
> ```

> [!warning] `lsof -i :80` — это **все** соединения на порту, не только «кто слушает»
> Команда покажет и слушающий сокет, и установленные соединения на 80-м. Чтобы получить **именно слушающие** сокеты:
> ```bash
> sudo lsof -iTCP -sTCP:LISTEN -P -n      # все слушающие TCP-порты + чьи они
> sudo lsof -iTCP:80 -sTCP:LISTEN -P -n   # конкретно кто слушает 80
> ```

## 🧰 Дополнение — что ещё полезно знать

| Команда | Что делает |
| :--- | :--- |
| `lsof -t -i:8080` | **только PID** (terse) — удобно в пайп: `kill $(lsof -t -i:8080)` |
| `lsof -c sshd` | открытые файлы по **имени процесса** (а не PID) |
| `lsof +D /var/log` | **рекурсивно** по всему каталогу (медленнее); `lsof /var/log` — только сам каталог |
| `lsof -a -u www-data -i` | **И-логика** (`-a`): файлы, что и у юзера, **и** сетевые |
| `lsof -u ^root` | исключить пользователя (`^` = «кроме») |
| `lsof -i @192.168.1.10` | соединения с/к конкретным хостом |
| `lsof -i :1-1024` | диапазон портов |
| `lsof -r2 -i :443` | **повтор** каждые 2 сек (мониторинг в реальном времени) |
| `lsof /dev/sdb1` | кто держит **раздел/диск** (если не размонтируется) |

> [!tip] Восстановить удалённый, но ещё открытый файл
> Классика: процесс пишет в лог/файл, файл удалили (`rm`), место **не освободилось**, потому что дескриптор открыт. Пока процесс жив — данные можно вытащить через `/proc`:
> ```bash
> sudo lsof +L1                       # найди PID и номер FD удалённого файла
> sudo cp /proc/<PID>/fd/<FD> ~/recovered.log   # копия содержимого
> ```
> Так же освобождают место: место вернётся только когда процесс закроет дескриптор (или будет перезапущен).

## 🛡️ Разбор «подозрений на взлом» (blue-team)

> [!note] Типовой триаж через lsof
> - **Малварь, удалившая свой бинарник** (чтобы спрятаться), но работающая: `sudo lsof +L1 | grep -i '/tmp\|/dev/shm\|(deleted)'` → подозрительные `(deleted)`-исполняемые с активным PID; копию для анализа — из `/proc/<PID>/fd/`.
> - **Левые сетевые соединения:** `sudo lsof -i -P -n` → что и куда стучит, особенно процессы из `/tmp`, `/dev/shm`, без пути.
> - **Кто читает чувствительное:** `sudo lsof -n /etc/shadow ~/.ssh/` — неожиданные процессы на ключах/секретах.
> - Это **триаж, а не вердикт**: PID можно подделать, руткит уровня ядра может прятать процессы от lsof. Для серьёзного инцидента — изоляция хоста и образ памяти/диска.

## 💻 Установка на твоих системах

| Система | Установка |
| :--- | :--- |
| **Gentoo** | `emerge sys-process/lsof` (часто уже стоит) |
| **Debian / Ubuntu** | `sudo apt install lsof` |
| **Arch** | `sudo pacman -S lsof` |
| **Entware / RT-AX56U** | `opkg install lsof` — полноценный lsof есть в репозитории под `armv7/aarch64` (в **BusyBox** своего `lsof` почти нет/он урезан). На роутере особенно полезно: **`lsof /opt`** или `lsof /tmp/mnt/...` — найти, **какой процесс не даёт отмонтировать USB-диск** перед извлечением. |

## 💡 Кому полезно

- **Админам/девопсам:** «кто занял порт», «почему не размонтируется», «что ест место после `rm`».
- **Безопасникам:** быстрый триаж хоста при инциденте (удалённые бинарники, странные коннекты).
- **Альтернативы под конкретную задачу:** для сокетов быстрее `ss -tulpn`; «кто держит файл/каталог» лаконично делает `fuser -v <файл>` / освобождает `fuser -k`.
