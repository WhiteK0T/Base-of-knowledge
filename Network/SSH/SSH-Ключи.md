## Что такое SSH-ключи

SSH-ключи — это пара криптографических ключей для безопасного подключения к серверам, Git-репозиториям (GitHub, GitLab) и другим удалённым системам **без ввода пароля**. Закрытый (приватный) ключ остаётся на вашем компьютере, а открытый (публичный) размещается на сервере в файле `~/.ssh/authorized_keys`.

> [!info] Связанная заметка
> Про разбор парольной аутентификации см. [Как узнать пароль user-a, который подключается к серверу по ssh](../../Linux/Security/Как%20узнать%20пароль%20user-a,%20который%20подключается%20к%20серверу%20по%20ssh.md).

## 📊 Актуальные типы SSH-ключей

| Тип ключа | Статус | Длина / Кривая | Безопасность | Производительность | Совместимость | Рекомендация |
|-----------|--------|----------------|--------------|--------------------|---------------|---------------|
| **Ed25519** | ✅ Актуальный (рекомендуемый) | Кривая 25519 (256 бит) | Отличная (128-бит) | Высокая (самый быстрый) | Высокая (OpenSSH 6.5+, GitHub/GitLab) | 🔥 Лучший выбор для новых систем |
| **Ed25519-SK** | ✅ Актуальный (для FIDO2) | Кривая 25519 | Отличная + 2FA | Высокая | Ограниченная (требуется FIDO2-ключ) | 🔐 Максимальная защита с аппаратным токеном |
| **RSA (4096 бит)** | ⚠️ Для совместимости | ≥3072, лучше 4096 бит | Хорошая | Низкая (медленный) | Максимальная (работает везде) | 📌 Резервный вариант для старых систем |
| **RSA (2048 бит)** | ⚠️ Устаревает | 2048 бит | Средняя | Средняя | Высокая | 🚫 Не рекомендуется для новых ключей |
| **ECDSA** | ⚠️ Не рекомендуется | NIST P-256/384/521 | Средняя | Хорошая | Высокая (OpenSSH 5.7+) | 🚫 Лучше использовать Ed25519 |
| **ECDSA-SK** | ⚠️ Ограниченно | NIST P-256 | Средняя + 2FA | Хорошая | Ограниченная | 🔘 Только если FIDO2-ключ не поддерживает Ed25519 |
| **DSA** | ❌ Устаревший | 1024 бит | Низкая | Низкая | Низкая | ❌ Запрещён к использованию |

### Пояснения к таблице

**Ed25519** — современный алгоритм на эллиптических кривых: высокая безопасность при малой длине ключа и отличной производительности. Сегодня это рекомендуемый стандарт для большинства задач; многие организации полностью переходят на него, отказываясь от RSA.

**Ed25519-SK** — версия Ed25519 для аппаратных токенов безопасности (например, YubiKey) с поддержкой протокола FIDO2. Обеспечивает двухфакторную аутентификацию: для входа нужен и сам ключ, и физическое устройство.

**RSA 4096 бит** — классический алгоритм, остаётся хорошим выбором там, где Ed25519 может не поддерживаться. Высокий уровень безопасности, но работает медленнее и даёт более длинные ключи, чем современные эллиптические алгоритмы. Размеры 1024 и 2048 бит считаются устаревающими.

**ECDSA** — более ранняя реализация эллиптической криптографии: сложнее в безопасной реализации и использует кривые NIST с не полностью прозрачным происхождением параметров (теоретический риск бэкдоров). Предпочтительнее Ed25519.

**DSA** — полностью устаревший алгоритм. OpenSSH прекратил его поддержку ещё в 2015 году; использовать DSA-ключи в современных системах нельзя.

## 🛠️ Как создать SSH-ключ

Для генерации ключей используется утилита `ssh-keygen`, встроенная в Linux, macOS и современные версии Windows (PowerShell / CMD).

### Шаг 1. Генерация ключа (рекомендуемый способ — Ed25519)

```bash
ssh-keygen -t ed25519 -C "ваш_email@example.com"
```

Флаг `-C` добавляет комментарий (обычно email или имя хоста), чтобы потом идентифицировать ключ.

Если нужна максимальная совместимость со старыми системами — RSA:

```bash
ssh-keygen -t rsa -b 4096 -C "ваш_email@example.com"
```

Для наивысшей защиты с аппаратным токеном — Ed25519-SK:

```bash
ssh-keygen -t ed25519-sk
```

### Шаг 2. Ответы на вопросы системы

1. **Enter file in which to save the key** — нажмите `Enter`, чтобы сохранить в расположение по умолчанию (`~/.ssh/id_ed25519`).
2. **Enter passphrase (empty for no passphrase)** — **настоятельно рекомендуется** задать надёжную парольную фразу. Это зашифрует закрытый ключ: даже если файл украдут, без passphrase им не воспользоваться.

### Шаг 3. Проверка созданных файлов

В папке `~/.ssh/` появятся два файла:

- `id_ed25519` (или `id_rsa`) — **ЗАКРЫТЫЙ (ПРИВАТНЫЙ) КЛЮЧ**. Его **никому и никогда** нельзя показывать или передавать.
- `id_ed25519.pub` (или `id_rsa.pub`) — **ОТКРЫТЫЙ (ПУБЛИЧНЫЙ) КЛЮЧ**. Именно его содержимое копируют на сервер или в настройки GitHub/GitLab.

## 🚀 Как использовать созданный ключ

### Способ А. Автоматическая загрузка на сервер (Linux/macOS)

Утилита `ssh-copy-id` сама добавит публичный ключ в `~/.ssh/authorized_keys` на сервере:

```bash
ssh-copy-id username@ip_адрес_сервера
```

Потребуется один раз ввести пароль пользователя на сервере.

### Способ Б. Ручное добавление (или для GitHub/GitLab)

1. Выведите содержимое публичного ключа:
   ```bash
   cat ~/.ssh/id_ed25519.pub
   ```
2. Скопируйте **всю** строку (начинается с `ssh-ed25519`, заканчивается вашим комментарием/email).
3. Вставьте её:
   - в файл `~/.ssh/authorized_keys` на целевом сервере (каждый ключ с новой строки);
   - или в раздел *SSH Keys* в настройках аккаунта GitHub / GitLab / Bitbucket.

## 🔒 Правила безопасности

- 🚫 **Никогда не делитесь закрытым ключом** (файлом без расширения `.pub`).
- 🔑 **Всегда защищайте закрытый ключ passphrase.** Чтобы не вводить её каждый раз, используйте `ssh-agent` (в macOS встроен в связку ключей, в Linux ключ добавляется командой `ssh-add ~/.ssh/id_ed25519`).
- 📁 Правильно выставляйте права доступа:
  ```bash
  chmod 700 ~/.ssh
  chmod 600 ~/.ssh/id_ed25519
  chmod 644 ~/.ssh/id_ed25519.pub
  ```
- 🔁 **Регулярно ротируйте ключи**, особенно если сотрудник увольняется или есть подозрение на компрометацию устройства. Старые публичные ключи удаляйте из `authorized_keys`.
- 🗑️ Избегайте создания ключей типов DSA, ECDSA и RSA короче 2048 бит.

## 🔬 Будущее: постквантовая криптография

Начиная с **OpenSSH 10.0** (апрель 2025), алгоритм обмена ключами `mlkem768x25519-sha256` стал новым стандартом по умолчанию. Эти алгоритмы устойчивы к атакам с использованием квантовых компьютеров и защищают от угрозы «собрать сейчас — расшифровать позже».

> [!warning] С OpenSSH 10.1 появляется предупреждение, если соединение не поддерживает постквантовые алгоритмы. Рекомендуется обновлять клиентские и серверные части SSH до актуальных версий.

## 💡 Краткие рекомендации

- 🚀 **По умолчанию используйте Ed25519:** `ssh-keygen -t ed25519`
- 📋 Нужна совместимость со старыми системами — RSA 4096: `ssh-keygen -t rsa -b 4096`
- 🛡️ Наивысшая защита — аппаратный токен с Ed25519-SK: `ssh-keygen -t ed25519-sk`
- 🔑 Всегда защищайте закрытый ключ надёжной passphrase.
