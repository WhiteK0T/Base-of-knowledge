# 🌿 git reflog — журнал перемещений HEAD

`git reflog` показывает **локальную** историю того, куда двигались `HEAD` и ветки: каждый коммит, переключение, [`reset`](reset.md), [`rebase`](rebase.md), [`merge`](merge.md), `amend`. Это «чёрный ящик» репозитория и главный **спасательный круг**: через него восстанавливают коммиты, пропавшие из обычного [`git log`](log.md).

```bash
git reflog
```

```text
a1b2c3d HEAD@{0}: commit: добавить парсер
9f8e7d6 HEAD@{1}: reset: moving to HEAD~1
1234abc HEAD@{2}: checkout: moving from main to feature
```

Слева — хеш, затем `HEAD@{N}` (позиция N шагов назад) и описание операции.

## 🧬 Чем reflog отличается от log

| | `git log` | `git reflog` |
| :--- | :--- | :--- |
| Что показывает | историю **коммитов** ветки (по родителям) | историю **движений** `HEAD`/веток |
| Видит «оборванные» коммиты | нет | **да** (даже после reset/rebase) |
| Где живёт | в графе коммитов (общий) | **локально** в `.git/logs/` (не пушится) |
| Срок жизни | пока есть коммит | по умолчанию ~90 дней, потом GC |

> [!info] reflog локальный и временный
> Reflog хранится только у тебя в `.git/logs/`, **не передаётся** при push/clone и постепенно очищается (`gc.reflogExpire`, по умолчанию 90 дней; недостижимые — 30). Поэтому это инструмент «здесь и сейчас» для своей машины, а не история проекта.

## 📝 Базовое использование

```bash
git reflog                       # = git reflog show HEAD
git reflog show <ветка>          # reflog конкретной ветки
git reflog --date=iso            # с датами вместо HEAD@{N}
git show HEAD@{2}                # посмотреть состояние на 2 шага назад
git diff HEAD@{1} HEAD           # что изменилось с прошлой позиции
```

Синтаксис `HEAD@{N}` (по счёту) и `HEAD@{<время>}` (по времени) работает в большинстве команд:

```bash
git show 'HEAD@{yesterday}'
git log 'main@{2.days.ago}'
```

## 🛟 Восстановление потерянного — главный сценарий

### После неудачного `reset --hard`

```bash
git reflog                       # найти хеш состояния ДО reset (напр. HEAD@{1})
git reset --hard HEAD@{1}        # вернуться на него
```

### «Потерянные» коммиты после `rebase`

```bash
git reflog                       # найти вершину ветки ДО rebase
git switch -c recovered <hash>   # развернуть их в новую ветку
```

### Вернуть удалённую ветку

```bash
git reflog                       # найти последний коммит удалённой ветки
git branch restored <hash>       # пересоздать ветку на нём
```

> [!tip] Первый шаг при «всё пропало»
> Почти любую **закоммиченную** работу, потерянную из-за reset/rebase/amend/удаления ветки, видно в `git reflog`. Алгоритм: `git reflog` → найти нужный хеш → `git reset --hard <hash>` (вернуть текущую ветку) или `git branch/switch -c <имя> <hash>` (развернуть в новую). См. также [`git reset`](reset.md).

## ⚠️ Чего reflog НЕ спасёт

> [!warning] Только закоммиченное
> Reflog фиксирует движения **ссылок** (коммиты, reset, checkout…). Он **не** хранит незакоммиченные правки: то, что снёс [`git checkout`/`restore`](restore.md), [`git reset --hard`](reset.md) по незакоммиченным изменениям или [`git clean`](clean.md), в reflog **не попадает** — восстановить нельзя. Отсюда привычка коммитить/[`stash`](stash.md) перед рискованными операциями.

## 🧹 Управление reflog

```bash
git reflog expire --expire=now --all     # очистить (редко нужно вручную)
git reflog delete HEAD@{2}               # удалить конкретную запись
git gc                                    # сборка мусора удалит недостижимое после истечения reflog
```

> [!note] Почему «потерянные» коммиты ещё живы
> Даже исчезнув из веток, коммит остаётся объектом в `.git/objects`, пока на него ссылается reflog и не прошёл `git gc`. Reflog держит эти ссылки ~90 дней — окно, в которое всё ещё можно восстановиться.

## 🩺 Частые вопросы

- **Сделал `reset --hard`, пропали коммиты** — `git reflog`, найди `HEAD@{N}` до сброса, `git reset --hard HEAD@{N}`.
- **Запорол `rebase`** — либо `git rebase --abort` (если ещё идёт), либо через reflog найди прежнюю вершину и разверни в ветку.
- **Удалил ветку с нужными коммитами** — `git reflog` → `git branch <имя> <hash>`.
- **В reflog нет моих правок** — они не коммитились/не стэшились; reflog хранит только ссылки на коммиты.
- **reflog пуст на свежем клоне** — он локальный и не клонируется; у нового клона своя пустая история ссылок.

## 💎 Резюме

- `git reflog` — локальная история движений `HEAD`/веток (`HEAD@{N}`), не пушится, живёт ~90 дней.
- Главное применение — **восстановление** после reset/rebase/amend/удаления ветки.
- Алгоритм: `reflog` → хеш → `reset --hard <hash>` или `branch/switch -c <имя> <hash>`.
- Спасает **закоммиченное**; незакоммиченное (clean/restore/reset --hard по правкам) — нет.
