# 🌿 git revert — отменить коммит новым коммитом

`git revert` отменяет указанный коммит, создавая **новый** коммит с обратными изменениями. История при этом **не переписывается** — старый коммит остаётся на месте, поверх него ложится «анти-коммит». Это **безопасный** способ откатить уже опубликованные изменения.

```bash
git revert <commit>          # создать коммит, отменяющий <commit>
```

> [!info] revert добавляет, а не стирает
> `git revert` не удаляет коммит из истории — он **дописывает** новый коммит, который отменяет его эффект. Поэтому он безопасен на **общих** ветках: ничьи хеши не меняются, force-push не нужен. Этим он принципиально отличается от [`git reset`](reset.md), который двигает ветку назад.

## 🆚 revert против reset

| | `git revert` | `git reset` |
| :--- | :--- | :--- |
| Что делает | новый коммит с обратными правками | двигает указатель ветки назад |
| История | **сохраняется** (+1 коммит) | переписывается (коммиты «исчезают») |
| Публичные ветки | **безопасно** | опасно (нужен force-push) |
| Когда | отменить **запушенный** коммит | убрать **локальные** коммиты до публикации |

> [!tip] Простое правило
> Коммит **уже на сервере** и им пользуются → `git revert`. Коммит **только локально**, ещё не публиковался → можно `git reset`. На общих ветках (`main`) — всегда revert.

## 📝 Базовое использование

```bash
git revert <commit>                  # отменить один коммит (откроет редактор сообщения)
git revert HEAD                      # отменить самый последний коммит
git revert HEAD~2                    # отменить коммит «2 назад» (только его!)
git revert <commitA> <commitB>       # отменить несколько (по отдельному коммиту на каждый)
git revert <старый>..<новый>         # отменить диапазон коммитов
```

> [!note] revert HEAD~2 отменяет ОДИН коммит
> `git revert HEAD~2` отменяет именно коммит на 2 шага назад, **не** трогая два последних. Чтобы откатить «последние три» — задай диапазон `git revert HEAD~3..HEAD` (Git создаст по обратному коммиту на каждый, от новых к старым).

## ⚙️ Полезные опции

| Опция | Описание |
| :--- | :--- |
| **`--no-edit`** | не открывать редактор, взять авто-сообщение `Revert "..."` |
| **`-n` / `--no-commit`** | применить обратные правки в индекс, но **не** коммитить (несколько revert одним коммитом) |
| **`-e` / `--edit`** | редактировать сообщение (по умолчанию) |
| **`-m <номер>` / `--mainline`** | для **merge-коммита** — указать, какого родителя считать «основным» |
| **`--continue` / `--skip` / `--abort`** | управление при конфликтах/серии |
| **`-s` / `--signoff`** | добавить `Signed-off-by` |

## 🔀 Откат merge-коммита — `-m`

У merge-коммита **два родителя**, и Git не знает, к какому «вернуться». Нужно явно указать «основную» линию через `-m`:

```bash
git revert -m 1 <merge-commit>       # 1 = первый родитель (обычно та ветка, КУДА вливали)
```

> [!warning] Откат merge — особый случай
> `-m 1` оставляет первого родителя (ветку-приёмник, обычно `main`) и отменяет принесённое слиянием. Учти: повторно влить ту же ветку потом будет нельзя без хитростей (Git сочтёт её уже слитой) — нужно «reverт реверта» или rebase. Откатывать merge revert'ом стоит осознанно.

## ⚔️ Конфликты при revert

Если отменяемые строки уже менялись позже, возникнет конфликт:

```bash
git status                   # "Unmerged paths"
# правишь файлы, убираешь маркеры <<<< ==== >>>>
git add <файл>
git revert --continue        # завершить
# либо:
git revert --abort           # отменить операцию, вернуть как было
```

## ⚡ Частые сценарии

### Откатить плохой коммит на общей ветке

```bash
git revert <hash>            # создаёт «Revert "<сообщение>"», безопасно для push
git push
```

### Несколько revert одним коммитом

```bash
git revert -n <c1> <c2> <c3>     # применить обратные правки в индекс, без коммитов
git commit -m "Откатить серию изменений X"
```

### Передумал отменять

```bash
git revert --abort               # во время конфликта/серии
# уже закоммитил revert и хочешь вернуть? — «revert реверта»:
git revert <revert-commit>
```

## 🩺 Частые вопросы

- **`revert` или `reset`?** — запушенное/общее → `revert`; локальное/неопубликованное → можно `reset`.
- **«error: commit ... is a merge but no -m option was given»** — отменяешь merge-коммит: добавь `-m 1`.
- **Хочу отменить несколько коммитов одним** — `git revert -n <...>` + один `git commit`.
- **Вернуть отменённое** — сделать revert самого revert-коммита.
- **Случайно начал revert** — `git revert --abort` вернёт состояние до операции.

## 💎 Резюме

- `git revert <commit>` — отменяет коммит **новым** коммитом, история цела (безопасно для push).
- Для **общих/запушенных** веток — revert; для локальных — можно [`reset`](reset.md).
- Merge-коммит: `git revert -m 1 <merge>` (указать основного родителя).
- Серия одним коммитом: `git revert -n ...` + `git commit`.
- Конфликт: правишь → `add` → `--continue`; передумал → `--abort`.
