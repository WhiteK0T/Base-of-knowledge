# 🌿 git blame — кто и когда менял строку

`git blame` показывает для **каждой строки** файла: в каком коммите она появилась, кто автор и когда. Помогает понять «почему здесь так» — найти коммит (и его сообщение/контекст), который ввёл конкретную строку.

```bash
git blame file.py
```

Вывод — построчно:

```text
a1b2c3d4 (WhiteK0T 2026-05-10 12:30:00 +0300  12) def parse(line):
e5f6a7b8 (WhiteK0T 2026-06-01 09:15:00 +0300  13)     return line.strip()
```

Слева: короткий хеш коммита · автор · дата · номер строки · сам текст.

## 📝 Частые формы

```bash
git blame file.py                    # весь файл
git blame -L 10,20 file.py           # только строки 10–20
git blame -L 10,+5 file.py           # 5 строк начиная с 10-й
git blame <commit> -- file.py        # каким файл был на этом коммите
git blame -L '/def parse/',+10 file.py   # от строки с регуляркой
```

> [!tip] Не «обвинение», а навигация
> Нашёл подозрительную строку → `git blame -L` по ней → берёшь хеш → [`git show <hash>`](show.md), чтобы увидеть **весь** коммит, его сообщение и контекст изменения. blame — это точка входа в историю строки, а не способ «найти виноватого».

## ⚙️ Полезные опции

| Опция | Описание |
| :--- | :--- |
| **`-L <начало>,<конец>`** | ограничить диапазоном строк (или `<n>,+<k>`, или `/regex/`) |
| **`-w`** | игнорировать изменения только в пробелах (не сваливать на форматирование) |
| **`-M`** | распознавать **перемещение** строк внутри файла |
| **`-C`** | распознавать **копирование** строк из других файлов |
| **`-C -C -C`** | агрессивнее искать происхождение скопированных строк |
| **`--since=<дата>`** | ограничить глубину истории |
| **`-l`** | показывать полный (длинный) хеш |
| **`-e` / `--show-email`** | показывать e-mail вместо имени |
| **`--first-parent`** | идти только по основной линии (игнорировать ветки слияний) |

> [!note] `-w`, `-M`, `-C` — против ложного авторства
> Без них blame «припишет» строку тому, кто всего лишь переформатировал отступы или перенёс блок. `-w` игнорирует пробелы, `-M` ловит перемещения, `-C` — копирование. Для честной картины происхождения часто используют `git blame -w -M -C file.py`.

## 🕳️ «Проигнорировать» массовые правки — ignore-revs

Коммит сплошного форматирования (prettier, `gofmt`) ломает blame — всё указывает на него. Можно его игнорировать:

```bash
git blame --ignore-rev <hash> file.py
# или завести файл со списком «шумных» коммитов:
git config blame.ignoreRevsFile .git-blame-ignore-revs
```

GitHub тоже читает `.git-blame-ignore-revs` и пропускает эти коммиты в веб-blame.

## ⚡ Частые сценарии

```bash
git blame -L 42,42 app.py            # кто ввёл строку 42
git blame -w -M -C util.py           # честное авторство без шума форматирования
git log -L 40,60:app.py              # ЭВОЛЮЦИЯ строк 40–60 (полные диффы, см. log)
```

> [!tip] blame показывает «кто сейчас», log -L — «как менялось»
> `git blame` даёт **текущего** автора каждой строки. Чтобы увидеть **всю историю** изменений конкретных строк (со всеми диффами) — [`git log -L <start>,<end>:file`](log.md).

## 🩺 Частые вопросы

- **Все строки указывают на один коммит форматирования** — `git blame --ignore-rev <hash>` или настрой `blame.ignoreRevsFile`.
- **Строку «приписали» не тому** — добавь `-w -M -C` (пробелы/перемещение/копирование).
- **Нужен весь коммит, а не только строка** — возьми хеш из blame → [`git show <hash>`](show.md).
- **Как менялась строка во времени** — `git log -L` (не blame).
- **Хочу состояние на дату/коммит** — `git blame <commit> -- file`.

## 💎 Резюме

- `git blame <файл>` — для каждой строки: коммит, автор, дата.
- Диапазон: `-L 10,20`; честное авторство: `-w -M -C`.
- Шумные форматирующие коммиты: `--ignore-rev` / `blame.ignoreRevsFile`.
- Полный коммит строки: [`git show <hash>`](show.md); история строк: [`git log -L`](log.md).
