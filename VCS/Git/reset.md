# 🌿 git reset — откат указателя ветки и индекса

`git reset` **перемещает указатель текущей ветки** на другой коммит и (опционально) приводит к нему индекс и рабочее дерево. Это главный инструмент «отменить локально»: снять коммиты, разиндексировать файлы, выбросить правки. Команда мощная и **частично необратимая** — важно понимать три её режима.

> [!danger] reset переписывает локальную историю
> `git reset` двигает **ветку назад**, «забывая» коммиты после новой позиции. На **уже опубликованных** (`git push`) коммитах это ведёт к расхождению с сервером — там вместо `reset` используй [`git revert`](commit.md) (создаёт обратный коммит, не переписывая историю). `reset` — для **локальных, ещё не запушенных** изменений.

## 🧬 Три дерева Git

Чтобы понять `reset`, нужно держать в голове три состояния (см. [`git add`](add.md)):

| Дерево | Что это |
| :--- | :--- |
| **HEAD** | последний коммит (на что указывает ветка) |
| **Индекс** (staging) | что уйдёт в следующий коммит |
| **Рабочее дерево** | реальные файлы на диске |

`git reset` умеет «откатить» эти три уровня на разную глубину — этим и отличаются его режимы.

## ⚙️ Три режима — `--soft` / `--mixed` / `--hard`

`git reset [<режим>] <commit>` двигает ветку на `<commit>`, а режим решает, **что ещё** подтянуть к этой точке:

| Режим | HEAD (ветка) | Индекс | Рабочее дерево | Итог |
| :--- | :---: | :---: | :---: | :--- |
| **`--soft`** | ↩ двигает | без изменений | без изменений | коммиты сняты, **все их правки лежат в индексе** (готовы к новому коммиту) |
| **`--mixed`** (по умолчанию) | ↩ двигает | ↩ сбрасывает | без изменений | коммиты сняты, правки **в рабочем дереве**, индекс пуст (нужен `git add`) |
| **`--hard`** | ↩ двигает | ↩ сбрасывает | ↩ сбрасывает | коммиты и **все правки уничтожены** — рабочее дерево как в `<commit>` |

```text
            HEAD/ветка   индекс   рабочее дерево
--soft        ←──            ·            ·          (правки → staging)
--mixed       ←──           ←──           ·          (правки → не staged)
--hard        ←──           ←──          ←──         (правки УДАЛЕНЫ)
```

> [!warning] `--hard` выбрасывает незакоммиченную работу безвозвратно
> `git reset --hard` **затирает** правки в индексе и рабочем дереве без подтверждения. Незакоммиченное не попадает даже в `reflog`. Перед `--hard` убедись, что ничего ценного не потеряешь (или сделай `git stash`).

## 📝 Частые сценарии

### Снять последний коммит, оставив правки

```bash
git reset --soft HEAD~1     # правки остаются в индексе (пересоберёшь коммит)
git reset HEAD~1            # = --mixed: правки в рабочем дереве, индекс чист
```

Типично, когда «закоммитил рано» или хочешь переписать последний коммит иначе (альтернатива `git commit --amend`).

### Полностью откатить N коммитов и правки

```bash
git reset --hard HEAD~3     # снять 3 коммита и ВСЕ изменения по ним
```

### Сбросить ветку на состояние сервера

```bash
git fetch origin
git reset --hard origin/master   # локальная ветка = в точности удалённая
```

### Разиндексировать файл (unstage)

```bash
git reset HEAD file.txt     # убрать file.txt из индекса, правки сохранить
```

> [!tip] Для файлов теперь есть `git restore`
> В современном Git (≥ 2.23) пофайловые операции вынесены в [`git restore`](add.md): `git restore --staged <файл>` (unstage) и `git restore <файл>` (выбросить правки). Это понятнее, чем перегруженный `git reset` — но `reset HEAD <файл>` по-прежнему работает.

## 🆚 reset против похожих команд

| Хочу… | Команда | Переписывает историю? |
| :--- | :--- | :--- |
| снять/сдвинуть **локальные** коммиты | `git reset` | да (только локально) |
| **отменить опубликованный** коммит | `git revert <commit>` | нет (добавляет обратный коммит) |
| убрать файл из **индекса** | `git restore --staged <файл>` (или `git reset HEAD <файл>`) | нет |
| **выбросить правки** в файле | `git restore <файл>` (или `git checkout -- <файл>`) | нет |
| удалить **неотслеживаемые** файлы | `git clean -fd` | нет |

> [!info] reset двигает ВЕТКУ, checkout/switch двигает HEAD
> `git reset <commit>` перемещает **указатель ветки** на коммит. `git checkout <commit>` / `git switch` перемещает только **HEAD** (просмотр другого состояния), не трогая ветку. Разные задачи: «переписать ветку» vs «сходить посмотреть».

## 🛟 Если откатил лишнее — reflog

Даже после `git reset --hard` **закоммиченные** ранее коммиты остаются в [`git reflog`](log.md) (~90 дней) — можно вернуться:

```bash
git reflog                       # найти хеш состояния ДО reset
git reset --hard HEAD@{1}        # вернуться на шаг назад
```

> [!danger] reflog спасает коммиты, но НЕ незакоммиченные правки
> Если `--hard` снёс изменения, которые **никогда не коммитились и не стэшились**, их в reflog нет — восстановить нельзя. Отсюда привычка: перед рискованным `--hard` делать `git stash` или коммит-черновик.

## 🩺 Частые вопросы

- **`reset` против `revert`** — `reset` для локальной, ещё не запушенной истории; `revert` — для уже опубликованной (безопасен для общих веток).
- **`reset --hard` ничего не сделал с untracked-файлами** — верно, он не трогает неотслеживаемые; для них `git clean -fd` (осторожно, тоже необратимо).
- **`HEAD~1` против `HEAD^`** — почти синонимы: `HEAD~1` = первый родитель на 1 шаг назад; `HEAD^` = первый родитель (`HEAD^2` — второй родитель merge-коммита).
- **Случайно сделал `reset --hard`** — сразу `git reflog`, найди прежний хеш, `git reset --hard <hash>` (для закоммиченного).

## 💎 Резюме

- `--soft` — снять коммиты, правки в **индекс**; `--mixed` (умолч.) — в **рабочее дерево**; `--hard` — **удалить** всё.
- Разиндексировать файл: `git reset HEAD <файл>` (или новее — `git restore --staged`).
- Опубликованное отменяй через `git revert`, **не** `reset`.
- `--hard` необратим для незакоммиченного → подстрахуйся `git stash`.
- Откатил лишнее → `git reflog` + `git reset --hard <hash>`.
