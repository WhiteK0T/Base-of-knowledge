# 🌿 git restore — восстановление файлов

`git restore` восстанавливает содержимое **файлов** в рабочем дереве и/или индексе из коммита или из индекса. Появился в Git **2.23** (2019), чтобы разгрузить перегруженные `git checkout` и `git reset`: одна команда отвечала и за переключение веток, и за откат файлов. `restore` берёт на себя именно **пофайловое восстановление**.

> [!info] restore работает с ФАЙЛАМИ, не с историей
> В отличие от [`git reset`](reset.md), `git restore` **не двигает указатель ветки** и не трогает коммиты — он лишь меняет содержимое файлов. Это его принципиальное отличие: «откатить файл», а не «переписать историю».

## 🧬 Две цели восстановления

У `restore` два «адресата», задаются флагами `--worktree` (`-W`, по умолчанию) и `--staged` (`-S`):

| Команда | Что делает | Источник по умолчанию |
| :--- | :--- | :--- |
| `git restore <файл>` | вернуть файл в **рабочем дереве** (выбросить незастейдженные правки) | индекс |
| `git restore --staged <файл>` | убрать файл из **индекса** (unstage), рабочее дерево не трогать | `HEAD` |
| `git restore --staged --worktree <файл>` | сбросить и индекс, и рабочее дерево | `HEAD` |
| `git restore --source=<commit> <файл>` | взять версию файла из конкретного коммита | указанный коммит |

```text
                  --source (откуда брать)
                          │
   HEAD / commit ──────► индекс ──────► рабочее дерево
              --staged ──┘     --worktree ──┘
                (-S)               (-W, по умолчанию)
```

## 📝 Частые сценарии

### Выбросить незастейдженные правки в файле

```bash
git restore file.txt            # вернуть file.txt к версии из индекса
git restore .                   # выбросить ВСЕ незастейдженные правки
git restore src/                # по каталогу
```

> [!danger] Это необратимо
> `git restore <файл>` **затирает** незакоммиченные и незаиндексированные правки — их не вернуть (в `reflog` они не попадают). Если не уверен — сначала `git stash` или посмотри, что потеряешь, через [`git diff`](diff.md).

### Снять файл из индекса (unstage)

```bash
git restore --staged file.txt   # файл выходит из staging, правки в дереве целы
```

Это «новый» способ вместо `git reset HEAD file.txt`. Правки при этом **сохраняются** в рабочем дереве — меняется только то, что уйдёт в коммит.

### Полностью откатить файл к последнему коммиту

```bash
git restore --staged --worktree file.txt   # и индекс, и рабочее дерево → как в HEAD
# короткий эквивалент с явным источником:
git restore --source=HEAD --staged --worktree file.txt
```

### Достать версию файла из старого коммита

```bash
git restore --source=HEAD~3 file.txt    # содержимое файла, каким оно было 3 коммита назад
git restore --source=<хеш> path/to/file
git restore --source=feature file.txt   # версия из другой ветки
```

> [!note] restore не переключает ветку
> Достать **один файл** из другой ветки — это `git restore --source=<ветка> <файл>` (текущая ветка не меняется). Полное переключение ветки — `git switch`, а не restore.

### Интерактивно, по кускам

```bash
git restore -p file.txt         # выбрать, какие hunk'и откатить (как git add -p наоборот)
```

## ⚙️ Ключевые опции

| Опция | Описание |
| :--- | :--- |
| **`-W` / `--worktree`** | восстановить рабочее дерево (по умолчанию) |
| **`-S` / `--staged`** | восстановить индекс (unstage); источник по умолчанию — `HEAD` |
| **`-s` / `--source=<rev>`** | откуда брать содержимое (коммит/ветка/тег); иначе индекс (для `-W`) или `HEAD` (для `-S`) |
| **`-p` / `--patch`** | интерактивно выбирать куски для восстановления |
| **`--ours` / `--theirs`** | при конфликте слияния взять свою / чужую сторону |
| **`-q` / `--quiet`** | без лишнего вывода |

## 🆚 restore против reset и checkout

| Задача | Современно (рекомендуется) | Старый способ |
| :--- | :--- | :--- |
| выбросить правки в файле | `git restore <файл>` | `git checkout -- <файл>` |
| unstage файла | `git restore --staged <файл>` | `git reset HEAD <файл>` |
| версия файла из коммита | `git restore --source=<rev> <файл>` | `git checkout <rev> -- <файл>` |
| переключить ветку | `git switch <ветка>` | `git checkout <ветка>` |
| сдвинуть **ветку**/снять коммиты | `git reset <commit>` | — |

> [!tip] Почему restore/switch лучше checkout
> Старый `git checkout` делал слишком разное (и ветки, и файлы) — отсюда путаница и опасные опечатки. `restore` (файлы) и `switch` (ветки) **разделили обязанности**: команда явно говорит о намерении. `reset` остаётся для операций над **указателем ветки** и историей.

## 🩺 Частые вопросы

- **`restore` или `reset`?** — `restore` для **содержимого файлов**; `reset` — когда нужно сдвинуть **ветку** / снять коммиты.
- **Случайно выбросил правки через `restore`** — увы, незакоммиченное не восстановить (нет в reflog). На будущее — `git stash` перед массовым откатом.
- **`restore .` не удалил новый файл** — `restore` работает с **отслеживаемыми**; неотслеживаемые убирает `git clean -fd`.
- **Конфликт после неудачного merge** — `git restore --ours/--theirs <файл>` выбирает сторону, либо `git restore --source=HEAD <файл>` для сброса.

## 💎 Резюме

- Выбросить правки в файле: `git restore <файл>` (необратимо!).
- Unstage: `git restore --staged <файл>` (правки целы).
- Откатить файл целиком к `HEAD`: `git restore -SW <файл>`.
- Версия из коммита/ветки: `git restore --source=<rev> <файл>`.
- Файлы → `restore`, ветки → `switch`, история/указатель → `reset`.
