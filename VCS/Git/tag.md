# 🌿 git tag — метки версий (релизы)

`git tag` ставит **именованную метку** на коммит — обычно для пометки релизов (`v1.0.0`). В отличие от [веток](branch.md), тег **не двигается**: он навсегда указывает на конкретный коммит, фиксируя «вот это — версия 1.0».

## 🧬 Тег против ветки

| | Тег | Ветка |
| :--- | :--- | :--- |
| Двигается при коммите | **нет** (фиксирован) | да (сдвигается за `HEAD`) |
| Назначение | пометить точку (релиз) | вести разработку |
| Где хранится | `.git/refs/tags/` | `.git/refs/heads/` |

> [!info] Тег — «закладка» на версию
> Тег навсегда привязан к одному коммиту. По нему удобно собирать релиз, сравнивать версии ([`git diff v1.0 v2.0`](diff.md)), вернуться к точному состоянию ([`git switch -d v1.0`](switch.md)).

## 🔖 Два типа тегов

| Тип | Команда | Что хранит |
| :--- | :--- | :--- |
| **Аннотированный** (annotated) | `git tag -a v1.0 -m "..."` | отдельный объект: автор, дата, сообщение, **можно подписать GPG** |
| **Лёгкий** (lightweight) | `git tag v1.0` | просто указатель на коммит, без метаданных |

> [!tip] Для релизов — аннотированные
> Аннотированный тег — полноценный объект с автором/датой/сообщением (и опционально подписью). Именно их используют для **публичных релизов**. Лёгкий тег — это «закладка на скорую руку» для личных нужд. Сомневаешься — ставь `-a`.

## 📝 Базовое использование

```bash
git tag                          # список тегов
git tag -l "v1.*"                # фильтр по шаблону
git tag -a v1.0.0 -m "Релиз 1.0.0"   # аннотированный тег на ТЕКУЩИЙ коммит
git tag v1.0.0                   # лёгкий тег
git tag -a v0.9.0 <commit> -m "..."  # тег на КОНКРЕТНЫЙ (старый) коммит
git show v1.0.0                  # детали тега + коммита
git tag -d v1.0.0                # удалить локально
```

## ⚙️ Основные опции

| Опция | Описание |
| :--- | :--- |
| **`-a`** | аннотированный тег |
| **`-m "<текст>"`** | сообщение тега (подразумевает `-a`) |
| **`-s` / `--sign`** | подписать тег GPG-ключом |
| **`-l <шаблон>` / `--list`** | список с фильтром (`git tag -l "v2.*"`) |
| **`-d <тег>`** | удалить тег локально |
| **`-f` / `--force`** | переустановить существующий тег на другой коммит |
| **`-n[<n>]`** | в списке показать `n` строк сообщения тега |
| **`--contains <commit>`** | теги, содержащие данный коммит |
| **`--points-at <commit>`** | теги, указывающие ровно на коммит |
| **`-v <тег>`** | проверить GPG-подпись тега |

## 🚀 Push тегов на сервер

`git push` **не отправляет** теги автоматически — их нужно пушить явно:

```bash
git push origin v1.0.0           # один тег
git push origin --tags           # все теги
git push --follow-tags           # коммиты + аннотированные теги (рекомендуется)
git push origin --delete v1.0.0  # удалить тег на сервере
```

> [!warning] Теги пушатся отдельно
> Частая ошибка: поставил тег, сделал `git push` — а тега на GitHub нет. Нужен `git push origin <тег>` / `--tags`. На GitHub именно push аннотированного тега создаёт основу для **Release**. Удобно приучиться к `git push --follow-tags`.

## 🔢 Семантическое версионирование (SemVer)

Теги релизов принято именовать по **[SemVer](https://semver.org/lang/ru/)**: `vMAJOR.MINOR.PATCH`.

| Часть | Когда увеличивать |
| :--- | :--- |
| **MAJOR** (`v2.0.0`) | несовместимые изменения API |
| **MINOR** (`v1.3.0`) | новая функциональность, обратно совместимо |
| **PATCH** (`v1.2.4`) | багфиксы, обратно совместимо |

Префикс `v` (`v1.0.0`) — распространённое соглашение. Предрелизы: `v1.0.0-rc.1`, `v1.0.0-beta`.

## ⚡ Частые сценарии

### Поставить релизный тег и опубликовать

```bash
git tag -a v1.2.0 -m "Релиз 1.2.0"
git push origin v1.2.0           # (или git push --follow-tags)
```

### Тег на старый коммит (забыл при релизе)

```bash
git tag -a v1.1.0 <hash-того-коммита> -m "Релиз 1.1.0"
git push origin v1.1.0
```

### Переместить ошибочный тег

```bash
git tag -f v1.2.0 <правильный-commit>    # переустановить локально
git push origin -f v1.2.0                # обновить на сервере (осторожно!)
```

> [!warning] Не передвигай опубликованные теги
> Если тег уже на сервере и им пользуются (CI, зависимости), перемещать его (`-f`) опасно — у других останется старая версия под тем же именем. Лучше выпустить **новый** тег (`v1.2.1`), чем переписывать существующий.

## 🩺 Частые вопросы

- **Тег не появился на GitHub** — `git push` не шлёт теги: `git push origin <тег>` / `--tags` / `--follow-tags`.
- **Аннотированный или лёгкий?** — для релизов `-a` (метаданные + возможность подписи); лёгкий — для личных пометок.
- **Вернуться к версии** — `git switch -d v1.0.0` (detached) или `git checkout v1.0.0`; собрать архив — `git archive v1.0.0`.
- **Сравнить версии** — `git diff v1.0.0 v2.0.0` (см. [diff](diff.md)).
- **Удалить везде** — локально `git tag -d v1.0.0`, на сервере `git push origin --delete v1.0.0`.

## 💎 Резюме

- Релизный тег: `git tag -a vX.Y.Z -m "..."` (аннотированный, по [SemVer](https://semver.org/lang/ru/)).
- Список/фильтр: `git tag`, `git tag -l "v1.*"`; детали: `git show <тег>`.
- Публикация: `git push origin <тег>` / `--tags` / `--follow-tags` (отдельно от коммитов!).
- Удалить: `-d` локально + `push --delete` на сервере.
- Опубликованные теги **не передвигай** — выпускай новый.
