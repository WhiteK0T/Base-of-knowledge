# 🌿 git rebase — перенос и переписывание коммитов

`git rebase` **переносит** коммиты текущей ветки так, будто они были сделаны поверх другой базы. В отличие от [`git merge`](merge.md), который объединяет истории merge-коммитом, rebase **переписывает** коммиты (новые хеши) и даёт **линейную** историю.

```bash
git switch feature
git rebase main          # перенести коммиты feature поверх свежей main
```

## 🧬 Как это работает

Rebase берёт коммиты твоей ветки **после** общего предка, временно откладывает их, перематывает ветку на новую базу и **заново накладывает** коммиты один за другим (каждый — с новым хешем).

```text
до:    A → B → C  (main)
        \
         D → E    (feature)

git rebase main (на feature):
       A → B → C  (main)
                \
                 D' → E'   (feature — те же изменения, НОВЫЕ коммиты)
```

> [!warning] rebase создаёт НОВЫЕ коммиты
> `D'`/`E'` — это **копии** с другими хешами, а не те же `D`/`E`. Поэтому rebase **переписывает историю**: на уже опубликованной (запушенной) ветке это ломает синхронизацию с сервером и ломает работу другим. См. «золотое правило» ниже.

## 🆚 rebase против merge

| | `git rebase` | `git merge` |
| :--- | :--- | :--- |
| История | линейная, без merge-коммитов | ветвистая, с merge-коммитом |
| Коммиты | переписывает (новые хеши) | сохраняет как есть |
| Безопасность | **нельзя** на опубликованном | безопасен на общих ветках |
| Когда | подровнять свою ветку до push | влить готовую ветку в `main` |

> [!danger] Золотое правило rebase
> **Не перебазируй коммиты, которые уже запушены и которыми пользуются другие.** Rebase меняет хеши — после него у других «разъедется» история. Безопасно перебазировать только **локальные**, ещё не опубликованные коммиты. Исключение — своя личная ветка, где force-push согласован.

## 📝 Базовое использование

```bash
git rebase main                  # перенести текущую ветку поверх main
git rebase --onto A B            # перенести коммиты после B на A (хирургически)
git rebase --continue            # продолжить после разрешения конфликта
git rebase --skip                # пропустить проблемный коммит
git rebase --abort               # отменить rebase, вернуть как было
```

## 🎛️ Интерактивный rebase — `-i`

Главная суперсила: **переписать серию своих коммитов** перед публикацией (объединить, переименовать, переставить, удалить).

```bash
git rebase -i HEAD~4             # редактировать последние 4 коммита
git rebase -i main               # все коммиты ветки относительно main
```

Откроется список коммитов (старые сверху) с командами:

| Команда | Действие |
| :--- | :--- |
| **`pick`** | оставить коммит как есть |
| **`reword`** (`r`) | оставить, но изменить сообщение |
| **`edit`** (`e`) | остановиться на коммите, чтобы доправить содержимое |
| **`squash`** (`s`) | слить с предыдущим, **объединив** сообщения |
| **`fixup`** (`f`) | слить с предыдущим, **отбросив** своё сообщение |
| **`drop`** (`d`) | удалить коммит |
| **(переставить строки)** | изменить порядок коммитов |

> [!tip] Автосжатие правок — `--autosquash`
> Если коммитил «заготовки» через [`git commit --fixup=<hash>`](commit.md), то `git rebase -i --autosquash <база>` сам расставит их `fixup` к нужным коммитам. Удобно для чистой истории перед PR. Можно включить по умолчанию: `git config --global rebase.autosquash true`.

## ⚙️ Полезные опции

| Опция | Описание |
| :--- | :--- |
| **`-i` / `--interactive`** | интерактивное редактирование серии коммитов |
| **`--onto <new> <old>`** | перенести диапазон на произвольную базу |
| **`--autosquash`** | авто-расстановка `fixup!`/`squash!` коммитов |
| **`--autostash`** | спрятать/вернуть незакоммиченные правки вокруг rebase |
| **`--continue` / `--skip` / `--abort`** | управление при конфликтах |
| **`--exec "<cmd>"`** | выполнить команду после каждого коммита (напр. прогнать тесты) |
| **`--rebase-merges`** | сохранить структуру merge-коммитов при переносе |

## ⚔️ Конфликты при rebase

Поскольку коммиты накладываются по одному, конфликт может возникнуть на **каждом**:

```bash
# Git останавливается на конфликтном коммите:
git status                   # "Unmerged paths"
# правишь файлы, убираешь маркеры <<<< ==== >>>>
git add <файл>
git rebase --continue        # наложить дальше
# либо:
git rebase --skip            # выкинуть этот коммит
git rebase --abort           # отменить весь rebase, вернуть исходную ветку
```

> [!note] rebase vs merge при конфликте
> В [merge](merge.md) конфликт решаешь **один раз**. В rebase — потенциально на **каждом** переносимом коммите (зато история чище). Для длинных расходящихся веток merge иногда практичнее.

## 🔄 Типовой рабочий цикл

```bash
git switch feature
git fetch origin
git rebase origin/main           # подровнять ветку под свежую main
# (разрешить конфликты, если есть)
git push --force-with-lease      # обновить свою ветку на сервере (история переписана)
```

> [!warning] После rebase — только `--force-with-lease`
> Перебазированную ветку нельзя запушить обычным `git push` (история разошлась). Используй [`git push --force-with-lease`](push.md) — он не затрёт чужие коммиты, появившиеся на сервере.

## 🩺 Частые вопросы

- **rebase или merge?** — rebase для **личной** уборки до публикации (линейность); merge для вливания готовой ветки в общую.
- **Запутался в rebase** — `git rebase --abort` всегда вернёт исходное состояние.
- **«потерял» коммиты после rebase** — они в [`git reflog`](log.md): найди прежний хеш, `git reset --hard <hash>`.
- **pull всё время плодит merge-коммиты** — `git pull --rebase` (или `pull.rebase=true`), см. [pull](pull.md).
- **Нужно слить кучу мелких коммитов в один** — `git rebase -i` с `squash`/`fixup` (или [`git merge --squash`](merge.md) при вливании).

## 💎 Резюме

- `git rebase main` — перенести свою ветку поверх свежей базы, линейная история.
- Переписывает коммиты (новые хеши) → **только локальные/неопубликованные**.
- `git rebase -i` — объединить/переименовать/переставить/удалить коммиты (`squash`/`fixup`/`reword`/`drop`).
- Конфликты: правишь → `add` → `--continue` (или `--skip`/`--abort`).
- После rebase публикуй через `git push --force-with-lease`.
