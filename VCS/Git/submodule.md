# 🌿 git submodule — вложенные репозитории

`git submodule` подключает **другой Git-репозиторий** как подкаталог внутри твоего, при этом родитель фиксирует **конкретный коммит** подмодуля (а не его содержимое). Так подключают общие библиотеки/зависимости, оставляя их историю отдельной.

## 🧬 Как это устроено

Родительский репозиторий хранит для подмодуля **не файлы, а ссылку**: путь + URL (в `.gitmodules`) + **зафиксированный хеш коммита** (как специальная запись-gitlink в дереве). То есть «суперпроект» помнит: «в `libs/foo` должен быть репозиторий X на коммите `abc123`».

| Файл/запись | Что хранит |
| :--- | :--- |
| **`.gitmodules`** | путь + URL каждого подмодуля (коммитится) |
| запись в дереве | **зафиксированный коммит** подмодуля (gitlink) |
| `.git/modules/` | сами объекты подмодуля внутри родителя |

> [!info] Родитель указывает на КОММИТ, а не на ветку
> Подмодуль «приколот» к конкретному коммиту. Обновления в подмодуле **не приходят сами** — нужно вручную обновить ссылку и закоммитить её в родителе. Это и плюс (воспроизводимость), и источник путаницы.

## 📝 Базовый рабочий цикл

### Добавить подмодуль

```bash
git submodule add <url> libs/foo        # подключить репозиторий в libs/foo
git commit -m "Добавить подмодуль foo"  # коммитит .gitmodules + ссылку на коммит
```

### Клонировать проект с подмодулями

```bash
git clone --recurse-submodules <url>     # клон + сразу подмодули
# если клонировал без флага:
git submodule update --init --recursive  # доинициализировать и скачать
```

> [!warning] Свежий клон без `--recurse-submodules` → пустые подмодули
> Каталоги подмодулей будут **пустыми**, пока не выполнишь `git submodule update --init --recursive`. Частая причина «проект не собирается после clone».

### Обновить подмодули до новых коммитов

```bash
git submodule update --remote           # подтянуть свежие коммиты из ИХ веток
git add libs/foo                        # зафиксировать новую ссылку
git commit -m "Обновить подмодуль foo"
```

## ⚙️ Основные подкоманды

| Команда | Описание |
| :--- | :--- |
| **`add <url> <путь>`** | добавить подмодуль |
| **`init`** | зарегистрировать подмодули из `.gitmodules` в `.git/config` |
| **`update`** | привести подмодули к зафиксированным коммитам |
| **`update --init --recursive`** | init + update для всех (в т.ч. вложенных) |
| **`update --remote`** | обновить до свежих коммитов их веток |
| **`status`** | состояние подмодулей (хеши, рассинхрон) |
| **`foreach '<cmd>'`** | выполнить команду в каждом подмодуле |
| **`sync`** | обновить URL в `.git/config` после смены в `.gitmodules` |
| **`deinit <путь>`** | «отключить» подмодуль (очистить рабочее дерево) |

## 🗑️ Удалить подмодуль

Удаление многошаговое:

```bash
git submodule deinit -f libs/foo         # 1. деинициализировать
git rm libs/foo                          # 2. убрать из индекса и .gitmodules
rm -rf .git/modules/libs/foo             # 3. вычистить внутренние объекты
git commit -m "Удалить подмодуль foo"
```

## ⚡ Полезные приёмы

```bash
# обновить все подмодули и спуллить их ветки
git submodule update --remote --merge

# выполнить команду в каждом подмодуле
git submodule foreach 'git checkout main && git pull'

# клонировать с подмодулями параллельно
git clone --recurse-submodules -j8 <url>

# автоматически инициализировать подмодули при checkout (глобально)
git config --global submodule.recurse true
```

> [!tip] Альтернативы подмодулям
> Подмодули **хрупки** (легко забыть `update`, рассинхронизировать ссылку). Если нужна просто «вставка кода» без отдельной истории — рассмотри `git subtree` (включает содержимое в родителя) или менеджер пакетов проекта. Подмодули хороши, когда важно держать историю зависимости **раздельно** и фиксировать точную версию.

## 🩺 Частые вопросы

- **Подмодули пустые после clone** — `git submodule update --init --recursive` (или клонируй с `--recurse-submodules`).
- **`git status` в родителе показывает «new commits» у подмодуля** — внутри подмодуля переключились на другой коммит; зафиксируй ссылку (`git add <путь>`) или верни подмодуль на нужный коммит.
- **Изменил код в подмодуле** — коммить **внутри** подмодуля и пушь его отдельно, потом в родителе `git add <путь>` + коммит новой ссылки.
- **Сменился URL подмодуля** — поправь `.gitmodules`, затем `git submodule sync`.
- **«detached HEAD» в подмодуле** — норма: `update` ставит подмодуль на зафиксированный коммит, не на ветку.

## 💎 Резюме

- Подмодуль — вложенный репозиторий, родитель фиксирует его **конкретный коммит** (`.gitmodules` + gitlink).
- Клон: `git clone --recurse-submodules` (или `submodule update --init --recursive`).
- Обновить до свежего: `submodule update --remote` → `git add <путь>` → коммит.
- Обновления **не приходят сами** — ссылку двигаешь и коммитишь вручную.
- Хрупки: для простой «вставки кода» подумай про `subtree`/пакетный менеджер.
