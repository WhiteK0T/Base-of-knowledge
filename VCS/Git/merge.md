# 🌿 git merge — слияние веток

`git merge` объединяет историю двух [веток](branch.md): берёт изменения из указанной ветки и вливает их в **текущую**. Это основной способ собрать feature-ветку обратно в основную, сохранив историю обеих.

```bash
git switch main          # встать на ветку-ПРИЁМНИК
git merge feature        # влить feature в main
```

> [!info] merge вливает В текущую ветку
> `git merge feature`, стоя на `main`, добавит коммиты `feature` в `main`. Сама `feature` при этом **не меняется**. Всегда сначала перейди на ветку-приёмник ([`git switch`](switch.md)).

## 🧬 Два типа слияния

### Fast-forward (перемотка)

Если в приёмнике **не было** новых коммитов с момента ответвления, Git просто **двигает указатель** вперёд — без отдельного merge-коммита.

```text
до:    main → A
            \
             B → C  (feature)

после git merge feature (fast-forward):
       main ───────► C       (просто перемотка, новых коммитов нет)
```

### Трёхстороннее слияние (true merge)

Если **обе** ветки получили коммиты после расхождения, Git создаёт **merge-коммит** с двумя родителями, сводя три точки: общий предок + две вершины.

```text
до:    A → B → D  (main)
        \
         C → E    (feature)

после git merge feature:
       A → B → D → M   (M — merge-коммит, родители D и E)
        \         /
         C → E ──┘
```

## 📝 Базовое использование

```bash
git merge feature                # влить feature в текущую ветку
git merge origin/main            # влить состояние удалённой ветки
git merge --no-ff feature        # всегда создавать merge-коммит (даже если можно FF)
git merge --ff-only feature      # только перемотка, иначе отказать
git merge --squash feature       # собрать все правки feature в ОДИН коммит (без merge-коммита)
git merge --abort                # отменить незавершённое слияние (при конфликте)
```

## ⚙️ Основные опции

| Опция | Описание |
| :--- | :--- |
| **`--no-ff`** | всегда делать merge-коммит — сохраняет факт «здесь была ветка» |
| **`--ff-only`** | разрешить только fast-forward; при расхождении отказать |
| **`--squash`** | сплющить все коммиты ветки в один набор правок (коммит делаешь сам, родитель один) |
| **`--no-commit`** | слить, но остановиться перед merge-коммитом (дать проверить/доправить) |
| **`-m "<текст>"`** | задать сообщение merge-коммита |
| **`--abort`** | прервать слияние с конфликтом, вернуть состояние «до» |
| **`--continue`** | продолжить после разрешения конфликтов |
| **`-X ours` / `-X theirs`** | при конфликте автоматически предпочесть свою / чужую сторону |

> [!note] `--squash` против обычного merge
> `--squash` **не** создаёт merge-коммит и **не** записывает второго родителя — он просто складывает все изменения ветки в индекс одним пакетом, а коммит ты делаешь вручную. История остаётся линейной, но связь с исходной веткой теряется. Удобно, когда feature-ветка была «грязной» (много мелких коммитов).

## ⚔️ Конфликты слияния

Когда обе ветки изменили одни и те же строки, Git не может решить сам:

```text
CONFLICT (content): Merge conflict in file.txt
Automatic merge failed; fix conflicts and then commit the result.
```

В файле появляются маркеры:

```text
<<<<<<< HEAD
твоя версия (текущая ветка)
=======
их версия (вливаемая ветка)
>>>>>>> feature
```

Разрешение:

```bash
git status                   # список "Unmerged paths"
# отредактировать файлы, убрать маркеры, оставить нужное
git add file.txt             # пометить как разрешённый
git merge --continue         # (или git commit) — завершить слияние
# либо передумать:
git merge --abort            # вернуть всё как было до merge
```

> [!tip] Помощники при конфликтах
> - `git merge -X ours/theirs` — авто-выбор стороны для **конфликтующих** кусков (не путать с `--strategy`).
> - `git checkout --ours/--theirs <файл>` (или [`git restore --ours/--theirs`](restore.md)) — взять файл целиком с одной стороны.
> - `git mergetool` — открыть графический трёхпанельный разрешатель.

## 🆚 merge против rebase

| | `git merge` | `git rebase` |
| :--- | :--- | :--- |
| История | сохраняет как есть, добавляет merge-коммит | переписывает: коммиты «поверх» цели |
| Вид | ветвистый граф | линейный |
| Безопасность | не меняет существующие коммиты | **переписывает** хеши (нельзя на опубликованном) |
| Когда | влить готовую ветку в общую (`main`) | подровнять свою ветку перед push |

> [!warning] Золотое правило rebase
> `merge` безопасен для **общих** веток (не переписывает историю). `rebase` — для **локальной** уборки до публикации. Подробнее — [`git rebase`](rebase.md).

## 🩺 Частые вопросы

- **Хочу видеть, что ветка существовала** — `git merge --no-ff feature` (всегда merge-коммит).
- **Хочу линейную историю** — `git merge --ff-only` или `rebase` перед merge.
- **Куча мелких коммитов в feature** — `git merge --squash` + один свой коммит.
- **Запутался в конфликте** — `git merge --abort` вернёт состояние до слияния.
- **Случайно влил не то** — пока не запушено: `git reset --hard ORIG_HEAD` (Git запоминает позицию до merge).

## 💎 Резюме

- Слить: встать на приёмник (`git switch main`) → `git merge feature`.
- FF — простая перемотка; true merge — merge-коммит с двумя родителями.
- `--no-ff` сохраняет «след» ветки, `--squash` сплющивает в один коммит, `--ff-only` запрещает merge-коммиты.
- Конфликт: правишь → `git add` → `git merge --continue`; передумал → `--abort`.
- Для общих веток — merge; для локальной уборки — rebase.
