# 🔐 git-crypt — прозрачное шифрование файлов в репозитории

**git-crypt** ([github.com/AGWA/git-crypt](https://github.com/AGWA/git-crypt)) — инструмент для **прозрачного шифрования отдельных файлов** внутри обычного Git-репозитория. Помеченные файлы автоматически шифруются при `git add`/`commit`/`push` и расшифровываются при `checkout`/`pull` — **в рабочей копии они лежат открытым текстом, а в истории и на удалённом сервере хранятся как зашифрованные блобы** (AES-256-CTR с синтетическим IV). Idеально, чтобы держать в **публичном** репозитории секретные заметки (VPN/прокси, ключи, пароли) рядом с открытыми, не вынося их в отдельное хранилище. Версия **0.8.0** (сен 2025), лицензия GPL-3.

> [!info] Зачем это в нашем публичном вульте
> Этот репозиторий [рендерится на GitHub](../../README.md) и виден всем. git-crypt позволяет завести папку (например `Private/`) с заметками про VPN, прокси, конфиги — локально в Obsidian они читаются как обычно, а на GitHub лежат **нечитаемым шифротекстом**. Доступ к расшифровке — только у того, у кого есть GPG-ключ или симметричный ключ.

## 🧬 Как работает

git-crypt вешается как **Git clean/smudge-фильтр** (механизм `.gitattributes`):

```text
рабочая копия (открытый текст)
   │  git add  →  clean-фильтр  →  ШИФРУЕТ
   ▼
индекс / коммит / push  →  GitHub видит шифротекст
   │  checkout/pull  →  smudge-фильтр  →  РАСШИФРОВЫВАЕТ
   ▼
рабочая копия (открытый текст)
```

Какие файлы шифровать — задаётся паттернами в `.gitattributes` (синтаксис как у [`.gitignore`](gitignore.md)):

```gitattributes
Private/** filter=git-crypt diff=git-crypt
*.key      filter=git-crypt diff=git-crypt
secret.md  filter=git-crypt diff=git-crypt
```

> [!warning] `dir/*` ≠ поддиректории
> Чтобы зашифровать **всё поддерево**, нужен `Private/**`, а не `Private/*` (последний не захватит вложенные папки). И сам `.gitattributes` шифровать **нельзя** — он должен оставаться открытым, иначе Git не узнает, что файлы зашифрованы.

## 🔑 Два режима ключа

| Режим | Как раздаётся доступ | Кому удобно |
| :--- | :--- | :--- |
| **GPG** (`add-gpg-user`) | по **GPG-ключу**; ключ к репо хранится в самом репо, зашифрованный твоим публичным GPG | один владелец с GPG-ключом / команда |
| **Симметричный** (`export-key`) | один **файл-ключ**, который ты переносишь между машинами вручную (по защищённому каналу) | быстро, без GPG; но ключ надо безопасно передавать |

Для одного владельца с несколькими машинами (Gentoo/Debian/Arch) — оба варианта рабочие: **GPG** удобнее, если у тебя уже есть синхронизированный GPG-ключ; **симметричный ключ** проще, но файл-ключ нельзя класть в репо и надо аккуратно копировать на каждую машину.

### Как создать GPG-ключ

Если своего GPG-ключа ещё нет — создать (один раз, на доверенной машине):

```bash
gpg --full-generate-key      # выбрать: тип RSA/ECC, размер 4096, срок, имя/почта, пароль
gpg --list-secret-keys --keyid-format=long   # посмотреть ID ключа
```

В выводе берётся **ID ключа** (или email/отпечаток) — его и скармливаешь git-crypt:

```bash
git-crypt add-gpg-user 0xA1B2C3D4E5F60789   # выдать доступ этому ключу
```

git-crypt зашифрует ключ репозитория **твоим публичным GPG** и положит результат в `.git-crypt/` (это безопасно коммитить). Расшифровать сможет только тот, у кого есть **приватный** GPG-ключ. На каждой машине этот приватный ключ должен присутствовать (`gpg --import`), тогда после клонирования сработает `git-crypt unlock` без аргументов.

### Как создать симметричный файл-ключ

Без GPG: ключ репозитория выгружается в один файл (создаётся при `git-crypt init`):

```bash
git-crypt export-key ~/bok-git-crypt.key    # сохранить симметричный ключ в файл
```

Этим же файлом разблокируешь репо на любой машине после клонирования:

```bash
git-crypt unlock ~/bok-git-crypt.key
```

> [!danger] Файл-ключ = доступ ко всем секретам
> Не клади его в репозиторий и не теряй: кто получил файл — прочитал всё зашифрованное. Храни вне репо (менеджер паролей / зашифрованный носитель) и **сделай бэкап** — без него и без GPG данные не вернуть.

## 📦 Установка

git-crypt нужен **на каждой машине**, где репозиторий расшифровывается. Для GPG-режима ещё нужен `gnupg`.

| Система | Команда |
| :--- | :--- |
| **Gentoo** | `emerge dev-vcs/git-crypt` (+ `app-crypt/gnupg`) |
| **Debian / Ubuntu** | `sudo apt install git-crypt gnupg` |
| **Arch** | `sudo pacman -S git-crypt gnupg` (репо extra) |
| **Windows 10 / 11** | `choco install git-crypt` или `scoop install git-crypt`; GPG — [Gpg4win](https://gpg4win.org/) либо gpg из [Git for Windows](https://gitforwindows.org/). Есть нюансы — см. ниже |
| **Entware / RT-AX56U** | пакета **нет** в репозитории opkg — см. ниже |

> [!danger] На роутере git-crypt не нужен (и его там нет)
> В Entware-репозитории под armv7 пакета `git-crypt` нет (потребовалась бы ручная сборка). Но главное — **расшифровывать секреты VPN/прокси на роутере незачем**: ключ от всего вульта на устройстве, торчащем в сеть 24/7 — плохая идея. Держи расшифровку на доверенных десктопах; роутеру эти заметки ни к чему.

### 🪟 Windows 10/11 — нюансы

git-crypt на Windows работает, но нужен [Git for Windows](https://gitforwindows.org/) (команды удобнее выполнять из **Git Bash**), а сам git-crypt проще всего поставить пакетным менеджером:

```powershell
choco install git-crypt      # Chocolatey
# или
scoop install git-crypt      # Scoop (main bucket)
```

Для **GPG-режима** нужен gpg: [Gpg4win](https://gpg4win.org/) (даёт `gpg` + Kleopatra) или gpg, идущий в комплекте с Git for Windows. Импортируй **тот же приватный GPG-ключ**, что на Linux (`gpg --import secret.asc`), дальше `git-crypt unlock` — как на любой машине. Симметричный режим тоже работает: `git-crypt unlock C:\path\bok.key`.

> [!warning] Официальные Windows-бинарники отстают: последний — 0.7.0
> В [релизах](https://github.com/AGWA/git-crypt/releases) `.exe` собран только **до версии 0.7.0**; актуальный **0.8.0 выложен лишь под Linux**. `choco`/`scoop`, которые тянут готовый бинарь, дают версию **эпохи 0.7.0**. Для фиксов/форматов из 0.8.0 — собирай из исходников (MinGW/MSYS2) или, что для тебя проще, запускай git-crypt **внутри WSL2** (там обычный `apt install git-crypt` = свежая версия и нормальные Unix-права на ключ).

> [!danger] На Windows файлы ключей создаются БЕЗ ограничения прав
> Проект честно предупреждает: Windows-поддержка «less tested» и **не выставляет restrictive-права на файлы ключей** (по этой причине разработчик называет её непригодной для многопользовательской машины). На **общем** ПК ключ репозитория/симметричный ключ доступен шире, чем на Linux. На личном однопользовательском ПК — терпимо, но симметричный `*.key` всё равно держи **вне репозитория** (менеджер паролей / зашифрованный носитель), а не рядом с рабочей копией.

> [!caution] Коммить через CLI, а не через git-GUI
> README прямо отмечает, что git-crypt **ненадёжно работает с рядом сторонних git-GUI** (Atlassian SourceTree и т.п.). GUI-клиент может обойти clean/smudge-фильтр и **закоммитить секрет открытым текстом** — в публичном репо это утечка. На Windows добавляй и коммить зашифрованные файлы из **Git Bash / командной строки**, где git-crypt гарантированно в цепочке фильтров, и после первого коммита обязательно проверь шифротекст (`git show HEAD:Private/файл.md`).

## 🚀 Быстрый старт (сценарий «папка Private/»)

В корне репозитория. Шаги 1, 2 и 4 — общие; шаг 3 зависит от выбранного режима ключа.

```bash
# 1. инициализировать git-crypt в репозитории
git-crypt init

# 2. указать, что шифровать (создать/дополнить .gitattributes)
echo 'Private/** filter=git-crypt diff=git-crypt' >> .gitattributes
```

**Шаг 3 — выбрать режим ключа:**

```bash
# Вариант A — GPG: выдать доступ своему GPG-ключу
#   узнать ID: gpg --list-secret-keys --keyid-format=long
git-crypt add-gpg-user 0xA1B2C3D4E5F60789
```

```bash
# Вариант Б — без GPG (симметричный ключ): доступ уже есть у текущей
#   рабочей копии после `git-crypt init` — добавлять никого не нужно.
#   Ключ стоит лишь выгрузить в файл (для бэкапа и других машин):
git-crypt export-key ~/bok-git-crypt.key     # хранить ВНЕ репозитория!
```

```bash
# 4. положить секретные заметки и закоммитить (одинаково для обоих вариантов)
mkdir -p Private
$EDITOR "Private/VPN — конфиги и ключи.md"
git add .gitattributes Private/
git commit -m 'Private: add encrypted VPN notes'
git push
```

> [!note] В чём разница на этом шаге
> При **GPG** ты явно выдаёшь доступ ключу (`add-gpg-user` создаёт коммит в `.git-crypt/`). При **симметричном** режиме после `git-crypt init` текущая копия уже разблокирована — экспорт ключа нужен не для шифрования, а только чтобы потом `unlock` на других машинах и иметь бэкап.

После этого в `Private/` локально — обычный текст, а на GitHub эти `.md` — бинарный шифротекст. Проверить, что именно шифруется:

```bash
git-crypt status          # список файлов: encrypted / not encrypted
git-crypt status -e       # только зашифрованные
```

> [!tip] Убедись, что секрет реально зашифрован
> Сразу после первого коммита открой файл из `Private/` на GitHub (или `git show HEAD:Private/файл.md`) — должно быть **нечитаемо** (начинается с `GITCRYPT`). Если виден открытый текст — паттерн в `.gitattributes` не сработал; **считай секрет утёкшим** и меняй его.

## 💻 Расшифровка на другой машине (после clone)

```bash
git clone git@github.com-base:WhiteK0T/Base-of-knowledge.git
cd Base-of-knowledge

# GPG-режим: нужен импортированный приватный GPG-ключ → разблокировать
git-crypt unlock

# ИЛИ симметричный режим: сначала на исходной машине экспортировать ключ
#   git-crypt export-key ~/bok.key   (перенести по защищённому каналу!)
# затем на новой машине:
git-crypt unlock ~/bok.key
```

После `unlock` файлы расшифрованы в рабочей копии. `git-crypt lock` — снова зашифровать рабочую копию (например перед тем как оставить машину без присмотра).

## 🧰 Команды

| Команда | Действие |
| :--- | :--- |
| `git-crypt init` | включить git-crypt в репозитории (генерит ключ репо) |
| `git-crypt status` | какие файлы зашифрованы, а какие нет |
| `git-crypt add-gpg-user <ID>` | выдать доступ GPG-ключу (создаёт коммит) |
| `git-crypt unlock [keyfile]` | расшифровать рабочую копию (GPG или файл-ключ) |
| `git-crypt lock` | зашифровать рабочую копию обратно |
| `git-crypt export-key <файл>` | выгрузить симметричный ключ для переноса |

## ☠️ Критичные подводные камни (особенно для публичного репо)

> [!danger] Имена файлов и коммиты НЕ шифруются
> git-crypt шифрует **только содержимое**. Утекают наружу: **имена файлов**, сообщения коммитов, **размеры** файлов, факт изменения файла, цели симлинков. То есть `Private/NordVPN — логин и пароль.md` спрячет текст, но **само имя выдаст**, что и где лежит. Называй файлы нейтрально и не пиши секреты в commit-message.

> [!danger] Закоммитил до настройки `.gitattributes` — секрет в истории навсегда
> Если файл попал в коммит **раньше**, чем был добавлен паттерн в `.gitattributes`, его открытый текст останется в истории Git и уедет на GitHub. `.gitattributes` действует только на будущие коммиты. Правильный порядок: **сначала** `.gitattributes`, потом добавлять секреты. Если ошибся — переписывать историю (`git filter-repo`) и **менять все засветившиеся секреты**, считая их скомпрометированными.

> [!warning] Потерял ключ — потерял данные; отозвать доступ нельзя
> Восстановления нет: лишился GPG-ключа / файл-ключа — зашифрованные заметки не прочитать. **Делай бэкап ключа** отдельно от репо. И в git-crypt **нет отзыва доступа** (`revoke`): если симметричный ключ утёк, всё, что им шифровалось, надо считать раскрытым — поможет только смена секретов и переинициализация.

> [!note] На GitHub зашифрованные заметки не отрендерятся
> Это и есть цель — но помни: в `Private/` Markdown-превью, [Dataview](../../Obsidian%20Base%20Settings.md) и ссылки работают **только локально** в Obsidian. На GitHub там будет бинарь. Не клади в зашифрованную папку то, что должно быть видно публично.

## ⚖️ Альтернативы

| Инструмент | Подход | Чем отличается от git-crypt |
| :--- | :--- | :--- |
| **git-crypt** | прозрачный фильтр, шифрует файлы целиком | проще всего; имена файлов видны |
| **[SOPS](https://github.com/getsops/sops)** | шифрует **значения** в YAML/JSON/env (age/GPG/KMS) | structured-секреты, видна структура; не прозрачен |
| **[transcrypt](https://github.com/elasticdog/transcrypt)** | тот же clean/smudge, но на bash+OpenSSL | без бинаря, симметричный пароль; та же утечка имён |
| **git-secret / BlackBox** | GPG-обёртки над файлами | держат рядом `.gpg`-файлы, менее прозрачны |
| **[age](https://github.com/FiloSottile/age)** | просто файловый шифратор | не интегрирован в Git, шифруешь руками |

Для задачи «папка секретных заметок в Obsidian-вульте» **git-crypt — самый удобный**: прозрачно, ничего не меняет в рабочем процессе. SOPS лучше для структурированных конфигов/секретов в DevOps.

## ❓ FAQ — типичные ошибки

> [!question] После клонирования в `Private/` лежит «мусор» / `GITCRYPT…`
> Репозиторий ещё **заблокирован** — ты не расшифровал рабочую копию. Выполни `git-crypt unlock` (GPG) или `git-crypt unlock ~/bok.key` (файл-ключ). Без ключа содержимое останется шифротекстом — это норма.

> [!question] Закоммитил, а на GitHub файл всё равно открытым текстом
> Паттерн в `.gitattributes` не сработал на момент коммита. Проверь: `git-crypt status` (файл должен быть `encrypted`), правильный ли путь/glob (`Private/**`, а не `Private/*`), не закоммичен ли файл **раньше** правила. Уже утёкший открытый текст в истории — **меняй секрет** и переписывай историю (`git filter-repo`).

> [!question] `git-crypt status` пишет `not encrypted` для файла из `Private/`
> Либо файл добавлен до правила (Git помнит старый clean-фильтр) — передёрни: `git rm --cached <файл> && git add <файл>`; либо glob не совпал. После правок `.gitattributes` существующие файлы не перешифровываются сами.

> [!question] `git-crypt: error: encrypted file has been tampered with!`
> Зашифрованный файл правил кто-то **без** git-crypt (или сломался merge). Обычно — следствие правки шифротекста напрямую/конфликта слияния в зашифрованном файле; откати файл к рабочей версии и не редактируй блобы вручную.

> [!question] `gpg: decryption failed: No secret key` при `unlock`
> На этой машине нет **приватного** GPG-ключа, которому выдан доступ. Импортируй его (`gpg --import secret.asc`) — публичного недостаточно. Если ключ другой, доступ надо выдать заново: `git-crypt add-gpg-user` с этим ключом (на машине, где репо уже разблокирован).

> [!question] Забыл `.gitattributes` — могу ли просто переименовать/перешифровать задним числом?
> Нет. git-crypt не «дошифровывает» прошлое: всё, что уже в истории открытым текстом, считается раскрытым. Единственный честный путь — **сменить секрет** и при необходимости вычистить историю.

## 💡 Когда использовать

- Нужно хранить **секретные заметки** (VPN, прокси, токены, пароли) в **публичном** Git/Obsidian-репо рядом с открытыми — и чтобы локально они читались прозрачно.
- Несколько своих машин: один GPG-ключ → `unlock` после клонирования, и всё синхронизируется через тот же `git push/pull`.
- **Не** подходит, если важна тайна **имён файлов**/структуры — тогда выноси секреты в отдельный приватный репозиторий целиком.
