# 🌿 git commit — фиксация изменений

`git commit` сохраняет **проиндексированные** изменения (то, что добавлено в staging-область через `git add`) как новый снимок-коммит в истории репозитория. Это базовая операция Git: всё, что не закоммичено, живёт только в рабочем дереве и индексе и может быть потеряно.

## 🧬 Что такое коммит на самом деле

Коммит — это **не «разница» (diff), а полный снимок** состояния отслеживаемых файлов на момент фиксации (Git хранит их эффективно, переиспользуя неизменённые объекты). У каждого коммита есть:

| Атрибут | Что это |
| :--- | :--- |
| **SHA-1/SHA-256 хеш** | уникальный идентификатор коммита (`a1b2c3d…`), вычисляется из содержимого |
| **Родитель(и)** | ссылка на предыдущий коммит; у merge-коммита их **два и более**, у самого первого — ноль |
| **Author** | кто **написал** изменения (имя + e-mail + дата) |
| **Committer** | кто **применил** их (может отличаться от author — напр. при `rebase`, `am`, cherry-pick) |
| **Сообщение** | текст коммита (тема + тело) |
| **Дерево (tree)** | снимок структуры каталогов/файлов |

> [!info] Author ≠ Committer
> При `git rebase`, `cherry-pick` или применении патчей **author** сохраняется от исходного автора, а **committer** становится тем, кто выполняет операцию. Поэтому `git log` иногда показывает обе даты. Личность берётся из `user.name`/`user.email` (`git config`).

После коммита указатель ветки (напр. `master`) и `HEAD` сдвигаются на новый коммит — он становится «вершиной» ветки.

## 📝 Синтаксис и базовое использование

```bash
git commit [опции] [--] [<путь>...]
```

Без опций откроется редактор (`$GIT_EDITOR` / `core.editor`) для ввода сообщения:

```bash
git commit
```

Чаще указывают сообщение сразу через `-m`:

```bash
git commit -m "Исправить опечатку в README"
```

> [!note] Коммитится только проиндексированное
> `git commit` фиксирует **только то, что попало в индекс** через `git add`. Изменённый, но не добавленный файл в коммит не войдёт. Проверить, что именно уйдёт в коммит: `git status` и `git diff --staged` (синоним `--cached`).

## ⚙️ Основные опции и флаги

| Опция | Описание | Пример |
| :--- | :--- | :--- |
| **`-m "<текст>"`** | Сообщение прямо в команде, без редактора. Несколько `-m` → абзацы (тема, тело). | `git commit -m "Тема" -m "Тело"` |
| **`-a` / `--all`** | Автоматически добавить **все отслеживаемые** изменённые/удалённые файлы. **Новые (untracked) — не добавит.** | `git commit -am "Обновить конфиг"` |
| **`--amend`** | Переписать **последний** коммит (дополнить файлами / поправить сообщение). Переписывает историю. | `git commit --amend` |
| **`--no-edit`** | С `--amend` — не открывать редактор, оставить старое сообщение. | `git commit --amend --no-edit` |
| **`-p` / `--patch`** | Интерактивно выбрать **куски (hunks)** изменений в коммит. | `git commit -p` |
| **`--allow-empty`** | Коммит без изменений (напр. для триггера CI). | `git commit --allow-empty -m "trigger CI"` |
| **`-s` / `--signoff`** | Добавить строку `Signed-off-by:` (DCO в open-source). **Это не подпись, а пометка.** | `git commit -s -m "..."` |
| **`-S` / `--gpg-sign`** | **Криптографически подписать** коммит ключом GPG/SSH. | `git commit -S -m "..."` |
| **`-v` / `--verbose`** | Показать diff в окне редактора при наборе сообщения. | `git commit -v` |
| **`-F <файл>`** | Взять сообщение из файла (`-` = stdin). | `git commit -F msg.txt` |
| **`-C <commit>`** | Переиспользовать сообщение (и автора) другого коммита; `-c` — то же, но с редактированием. | `git commit -C HEAD@{1}` |
| **`--author="Имя <mail>"`** | Задать автора, отличного от себя. | `git commit --author="A <a@x>" -m "..."` |
| **`--date="<дата>"`** | Переопределить дату автора. | `git commit --date="2026-01-01 12:00" -m "..."` |
| **`--fixup=<commit>`** | Заготовка для автосжатия при `rebase -i --autosquash`. | `git commit --fixup=a1b2c3d` |
| **`-n` / `--no-verify`** | Пропустить хуки `pre-commit` / `commit-msg`. | `git commit -n -m "..."` |

> [!warning] `-S` (подпись) ≠ `-s` (signoff)
> **`-s`** дописывает текстовую строку `Signed-off-by: Имя <mail>` — это лишь декларация авторства/согласия с DCO, её может подделать кто угодно. **`-S`** ставит **настоящую криптоподпись** (GPG или SSH-ключ), которую проверяют через `git log --show-signature`. Не путать.

## 🔗 Связь индекса, рабочего дерева и коммита

```text
рабочее дерево  --(git add)-->  индекс (staging)  --(git commit)-->  репозиторий
   правки                         что уйдёт                 история коммитов
```

- `git diff` — рабочее дерево **против** индекса (что ещё не добавлено).
- `git diff --staged` — индекс **против** последнего коммита (что уйдёт в коммит).
- `git commit -a` — «срезает угол»: индексирует отслеживаемые правки и сразу коммитит, минуя ручной `git add` (но **не** трогает новые файлы).

## ✍️ Сообщение коммита

### Правило 50/72

- **Тема** (первая строка) — до **50 символов**, в **повелительном наклонении** («Добавить…», «Исправить…», англ. «Add…», «Fix…»), без точки в конце.
- Затем **пустая строка**.
- **Тело** — по желанию, строки до **72 символов**; объясняет **«почему»**, а не «что» (что — видно в `git diff`).

```text
Исправить переполнение буфера в парсере конфига

Старый код читал строку фиксированным буфером 256 байт и падал
на длинных значениях. Заменено на динамическое чтение.

Closes #142
```

### Conventional Commits

Машиночитаемый стандарт — упрощает changelog и SemVer:

```text
<тип>[область]: <краткое описание>

[тело]

[footer, напр. BREAKING CHANGE: ...]
```

Типы: `feat` (фича), `fix` (правка бага), `docs`, `refactor`, `test`, `chore`, `build`, `ci`, `perf`, `style`. Пример: `feat(auth): добавить вход через Google`.

> [!note] Стиль зависит от проекта
> Conventional Commits — удобный, но **не обязательный** стандарт. В личных репозиториях (как этот вольт) сообщения часто формата `Домен/Тема: что сделал`. Главное — единообразие внутри проекта.

### Footer-трейлеры

В конце сообщения, после пустой строки, идут «трейлеры» вида `Ключ: значение`:

```text
Co-authored-by: Имя <mail@example.com>
Signed-off-by: Имя <mail@example.com>
Closes #142
```

`Co-authored-by` GitHub показывает как соавторов коммита; `Closes/Fixes #N` автоматически закрывает issue при попадании в основную ветку.

## 🔄 Правка и спецсценарии

### Поправить последний коммит — `--amend`

```bash
git add забытый_файл
git commit --amend --no-edit       # дополнить коммит, сообщение оставить
git commit --amend -m "Новое сообщение"   # только переписать текст
```

> [!danger] amend переписывает историю
> `--amend` создаёт **новый** коммит с новым хешем вместо старого. Если старый коммит уже **отправлен** (`git push`), исправленный придётся пушить с `--force-with-lease`, а соавторам — синхронизироваться. На общих ветках — с осторожностью.

### Пустой коммит

```bash
git commit --allow-empty -m "ci: перезапуск пайплайна"
```

### Автосжатие правок — `--fixup`

```bash
git commit --fixup=a1b2c3d           # пометить как правку к коммиту a1b2c3d
git rebase -i --autosquash a1b2c3d~  # автоматически вмержить правку в него
```

### Отменить «не туда» закоммиченное

```bash
git reset --soft HEAD~1   # снять последний коммит, изменения остаются в индексе
git reset HEAD~1          # снять коммит, изменения в рабочем дереве (не в индексе)
git revert <commit>       # создать НОВЫЙ коммит, отменяющий указанный (безопасно для push)
```

> [!tip] revert против reset
> На **уже опубликованной** истории отменяй коммиты через `git revert` (он добавляет обратный коммит, не переписывая историю). `git reset` — для локальных, ещё не запушенных коммитов.

## 🪝 Хуки, влияющие на commit

- **`pre-commit`** — запускается до создания коммита; ненулевой код прерывает коммит (линтеры, форматтеры, секрет-сканеры).
- **`commit-msg`** — проверяет/правит файл сообщения (напр. валидация Conventional Commits).
- **`prepare-commit-msg`** — заполняет шаблон сообщения до открытия редактора.

Пропустить (осознанно): `git commit --no-verify`. Хуки лежат в `.git/hooks/` либо управляются менеджерами (`pre-commit`, `husky`, `lefthook`).

## 🩺 Частые проблемы

- **«nothing to commit, working tree clean»** — нечего коммитить: правки не сделаны или не добавлены в индекс (`git add`).
- **«Please tell me who you are»** — не задан `user.name`/`user.email`: `git config --global user.name "..."` и `... user.email "..."` (или локально без `--global`).
- **Закоммитил лишний файл** — `git reset --soft HEAD~1`, поправить индекс, перекоммитить (если не запушено).
- **Огромный коммит «на всё подряд»** — лучше дробить через `git add -p` / `git commit -p` на атомарные коммиты по смыслу.

## 💎 Резюме

- Быстрый коммит: `git commit -m "..."`.
- Все отслеживаемые правки сразу: `git commit -am "..."` (новые файлы — отдельно `git add`).
- Дисциплинированно: `git add -p` → осмотреть `git diff --staged` → `git commit`.
- Поправить последний (локальный) коммит: `git commit --amend`.
- Отменить опубликованный: `git revert`, а **не** `reset`.
