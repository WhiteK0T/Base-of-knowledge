# 🌿 git add — индексация изменений

`git add` переносит изменения из **рабочего дерева** в **индекс** (staging area) — промежуточную область, из которой потом собирается коммит. Это способ явно выбрать, **что именно** войдёт в следующий [`git commit`](commit.md), а что подождёт.

## 🧬 Зачем нужен индекс

Git — двухступенчатый: правки сначала **подготавливаются** (stage), потом **фиксируются** (commit). Индекс позволяет собрать осмысленный, атомарный коммит, даже если в рабочем дереве намешано несколько разных изменений.

```text
рабочее дерево  --(git add)-->  индекс (staging)  --(git commit)-->  репозиторий
   правки                       «что уйдёт»                  история
```

- `git diff` — рабочее дерево **против** индекса (что ещё **не** добавлено).
- `git diff --staged` (`--cached`) — индекс **против** последнего коммита (что **уйдёт** в коммит).
- `git status` — сводно: что отслеживается, что в индексе, что нет.

> [!info] Индекс — это не «список файлов», а снимок содержимого
> `git add file` фиксирует в индексе **текущее** состояние файла. Если после `add` снова поправить файл, новые правки **в индекс не попадут** — нужно повторить `git add`. Поэтому `git status` может показывать один файл одновременно и в «staged», и в «not staged».

## 📝 Синтаксис и базовое использование

```bash
git add [опции] [--] <путь>...
```

```bash
git add file.txt            # один файл
git add src/ docs/          # каталоги (рекурсивно)
git add .                   # всё в текущем каталоге и ниже
git add -A                  # все изменения во всём репозитории
```

## ⚙️ Основные опции и флаги

| Опция | Описание | Пример |
| :--- | :--- | :--- |
| **`<путь>`** | Добавить конкретные файлы/каталоги (новые + изменённые). | `git add src/main.py` |
| **`.`** | Добавить всё из **текущего каталога** и ниже. | `git add .` |
| **`-A` / `--all`** | Все изменения **во всём репозитории**: новые, изменённые **и удалённые**. | `git add -A` |
| **`-u` / `--update`** | Только **уже отслеживаемые** файлы (изменённые/удалённые). Новые **не добавит**. | `git add -u` |
| **`-p` / `--patch`** | Интерактивно выбрать **куски (hunks)** внутри файлов. | `git add -p` |
| **`-i` / `--interactive`** | Интерактивное меню индексации. | `git add -i` |
| **`-n` / `--dry-run`** | Показать, что **было бы** добавлено, ничего не меняя. | `git add -n .` |
| **`-f` / `--force`** | Добавить файл, даже если он в `.gitignore`. | `git add -f build/out.bin` |
| **`-v` / `--verbose`** | Показывать каждый добавляемый файл. | `git add -v .` |
| **`-e` / `--edit`** | Открыть итоговый diff в редакторе и вручную отредактировать, что застейджить. | `git add -e` |

> [!warning] `.` vs `-A` vs `-u` — главное различие
> - **`git add .`** — всё из **текущего** каталога вниз (зависит от того, где стоишь).
> - **`git add -A`** — всё **по всему** репозиторию, **включая удаления**, независимо от каталога.
> - **`git add -u`** — только **отслеживаемые** (изменённые + удалённые), **новые файлы игнорирует**.
>
> Классическая ловушка: `git add .` в современном Git (≥ 2.0) тоже учитывает удаления в своём поддереве, но `-u`/`-A` явнее по смыслу. Когда сомневаешься — `git add -n` сначала.

## ✂️ Точечная индексация — `git add -p`

Самый ценный режим для аккуратных коммитов: Git разбивает изменения на **куски (hunks)** и для каждого спрашивает, что делать.

```bash
git add -p
```

Основные ответы:

| Клавиша | Действие |
| :--- | :--- |
| **`y`** | застейджить этот кусок |
| **`n`** | пропустить этот кусок |
| **`s`** | разбить кусок на более мелкие (split) |
| **`e`** | вручную отредактировать кусок |
| **`q`** | выйти, не трогая остальные |
| **`a`** | застейджить этот и все следующие в файле |
| **`d`** | пропустить этот и все следующие в файле |
| **`?`** | справка |

> [!tip] Атомарные коммиты
> Если в одном файле смешаны фикс бага и рефакторинг — `git add -p` позволяет разнести их по **разным** коммитам. Это делает историю читаемой и упрощает `revert`/`bisect`.

## ↩️ Отменить индексацию (unstage)

`git add` обратим — снять файл из индекса (вернуть в «not staged»), **не теряя** правок:

```bash
git restore --staged file.txt   # современный способ (Git ≥ 2.23)
git reset HEAD file.txt          # классический способ
git reset                        # снять из индекса всё
```

> [!note] unstage ≠ откат правок
> `git restore --staged` / `git reset HEAD` убирают файл **из индекса**, но изменения в рабочем дереве остаются. Чтобы **выбросить** сами правки — `git restore file.txt` (или `git checkout -- file.txt`), и это **необратимо**.

## 🚫 Что add не добавит сам

- **Файлы из `.gitignore`** — игнорируются (обойти осознанно: `git add -f`).
- **Пустые каталоги** — Git не отслеживает каталоги, только файлы. Приём: положить файл-заглушку `.gitkeep`.
- **Новые файлы при `git add -u`** — `-u` работает только по уже отслеживаемым.

> [!tip] Проверить, что и почему игнорируется
> ```bash
> git status --ignored          # показать игнорируемые файлы
> git check-ignore -v build/    # какое правило .gitignore сработало
> ```

## 🩺 Частые проблемы

- **«Добавил, но `commit` всё равно пустой»** — правки сделаны **после** `git add`; повторить `git add` (индекс хранит снимок на момент добавления).
- **«Файл не добавляется»** — он в `.gitignore` (`git check-ignore -v <файл>`), либо используешь `git add -u` для нового файла.
- **«Случайно застейджил лишнее»** — `git restore --staged <файл>` (правки сохранятся).
- **Закоммитил то, что не нужно отслеживать** — добавить в `.gitignore` и убрать из индекса: `git rm --cached <файл>`.

## 💎 Резюме

- Конкретно: `git add <файл>`; всё в поддереве: `git add .`; всё с удалениями: `git add -A`.
- Только отслеживаемые: `git add -u`.
- Аккуратные коммиты: `git add -p` (по кускам) → `git diff --staged` → [`git commit`](commit.md).
- Снять из индекса: `git restore --staged <файл>` (правки целы).
- Глянуть «вхолостую»: `git add -n`.
