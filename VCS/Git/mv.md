# 🌿 git mv — переместить/переименовать файл

`git mv` перемещает или переименовывает отслеживаемый файл и **сразу проводит изменение через индекс** — эквивалент «переместить + [`git add`](add.md) старого и нового пути» одной командой.

```bash
git mv old.txt new.txt           # переименовать
git mv file.txt src/             # переместить в каталог
git commit -m "Переименовать old.txt → new.txt"
```

## 🧬 git mv = mv + add

```bash
git mv a b
# эквивалентно:
mv a b
git add a b        # проиндексировать исчезновение a и появление b
```

> [!info] Git не хранит «переименование» как операцию
> Внутри Git нет понятия «rename» — он хранит снимки содержимого. «Переименование» Git **выводит сам**, видя, что файл с тем же содержимым исчез в одном пути и появился в другом. `git mv` лишь удобно индексирует обе стороны; результат тот же, что и при ручном `mv` + `git add`.

## ⚙️ Полезные опции

| Опция | Описание |
| :--- | :--- |
| **`-f` / `--force`** | перезаписать существующий целевой файл |
| **`-k`** | пропускать ошибки (не падать, если что-то нельзя переместить) |
| **`-n` / `--dry-run`** | показать, что было бы сделано, не делая |
| **`-v` / `--verbose`** | показывать каждое перемещение |

## 🔍 Как rename виден в diff/log

После `git mv` и коммита Git показывает переименование, если содержимое почти не менялось:

```bash
git status                       # "renamed: old.txt -> new.txt"
git diff --staged -M             # пометит как rename (-M = find-renames)
git log --follow new.txt         # история файла ЧЕРЕЗ переименование
```

> [!tip] `--follow` — история сквозь переименования
> Обычный [`git log <файл>`](log.md) может оборваться на переименовании. `git log --follow <файл>` продолжает историю и до прежнего имени. Распознавание rename управляется порогом схожести (`-M`, по умолчанию ~50%).

## ⚡ Частые сценарии

```bash
git mv src/old_name.py src/new_name.py    # переименовать модуль
git mv *.txt docs/                        # переместить группу (shell раскроет маску)
git mv -n big-refactor/ new-place/        # сперва проверить
```

> [!note] Регистр имени на разных ФС
> На регистронезависимых ФС (macOS, Windows) переименование только регистра (`File.txt → file.txt`) Git может не заметить. Надёжно — через `-f` или промежуточное имя: `git mv File.txt tmp && git mv tmp file.txt`.

## 🩺 Частые вопросы

- **Переименовал обычным `mv`** — не страшно: `git add -A` (или `git add старое новое`) проиндексирует как rename; коммить.
- **rename не распознался в diff** — слишком много изменилось вместе с перемещением; Git покажет как delete+add. Используй `-M`/уменьши порог, либо коммить переименование **отдельно** от правок содержимого.
- **История оборвалась на переименовании** — `git log --follow <файл>`.
- **«destination exists»** — целевой файл есть: `git mv -f` (осознанно) или выбери другое имя.
- **Только регистр** — см. заметку выше про регистронезависимые ФС.

## 💎 Резюме

- `git mv <из> <в>` — переместить/переименовать + сразу проиндексировать.
- Эквивалент `mv` + `git add` (Git выводит rename из содержимого, отдельной операции нет).
- Проверка: `-n`; перезапись: `-f`; история сквозь rename: [`git log --follow`](log.md).
- Чисто переименуй **отдельным** коммитом от правок содержимого — так rename виден явно.
