# 🌿 git cherry-pick — перенести отдельный коммит

`git cherry-pick` применяет **изменения конкретного коммита** (из любой ветки) поверх текущей, создавая **новый** коммит с теми же правками, но другим хешем. «Выбрать вишенку» — взять один нужный коммит, не сливая всю ветку.

```bash
git switch main
git cherry-pick <commit>     # применить изменения <commit> поверх main
```

> [!info] cherry-pick копирует коммит
> Берётся **diff** указанного коммита и накладывается на текущую ветку как новый коммит (новый хеш, тот же author, новый committer). Исходный коммит остаётся на месте — ты делаешь его **копию** в другом месте истории. Идейно близко к [`rebase`](rebase.md), но точечно — по одному коммиту.

## 🎯 Когда применять

- **Бэкпорт фикса**: взять багфикс из `main` в релизную ветку, не таща туда всё остальное.
- **Достать один коммит** из чужой/экспериментальной ветки, которую не хочешь сливать целиком.
- **Спасти коммит** из ветки, которую собираешься удалить/переписать.
- **Перенести коммит**, по ошибке сделанный не в той ветке (cherry-pick на нужную + удалить из исходной через [`reset`](reset.md)/[`rebase`](rebase.md)).

## 📝 Базовое использование

```bash
git cherry-pick <commit>             # один коммит
git cherry-pick <c1> <c2> <c3>       # несколько (в указанном порядке)
git cherry-pick <старый>..<новый>    # диапазон (БЕЗ старого: старый не входит)
git cherry-pick <старый>^..<новый>   # диапазон ВКЛЮЧАЯ старый
```

> [!note] Границы диапазона
> `A..B` берёт коммиты **после** `A` и до `B` включительно — сам `A` **не** входит. Чтобы включить и `A`, используй `A^..B` (от родителя `A`). Та же логика диапазонов, что в [`git log`](log.md).

## ⚙️ Полезные опции

| Опция | Описание |
| :--- | :--- |
| **`-x`** | дописать в сообщение строку `(cherry picked from commit <hash>)` — след происхождения |
| **`-e` / `--edit`** | отредактировать сообщение коммита |
| **`-n` / `--no-commit`** | применить правки в индекс/дерево, но **не** коммитить (собрать несколько в один) |
| **`-s` / `--signoff`** | добавить `Signed-off-by` |
| **`-m <номер>` / `--mainline`** | cherry-pick **merge-коммита** — указать основного родителя |
| **`--ff`** | если можно — просто перемотать (fast-forward), без нового коммита |
| **`--continue` / `--skip` / `--abort` / `--quit`** | управление при конфликтах/серии |

> [!tip] Оставляй след с `-x`
> При бэкпортах удобно `git cherry-pick -x <hash>` — он добавит в сообщение `(cherry picked from commit …)`. Потом легко понять, откуда коммит пришёл, и не продублировать его.

## ⚔️ Конфликты

Если изменения не ложатся чисто (контекст уехал) — будет конфликт:

```bash
git status                   # "Unmerged paths"
# правишь файлы, убираешь маркеры <<<< ==== >>>>
git add <файл>
git cherry-pick --continue   # завершить
# либо:
git cherry-pick --skip       # пропустить текущий коммит (при серии)
git cherry-pick --abort      # отменить всё, вернуть исходное состояние
```

## ⚡ Частые сценарии

### Бэкпорт фикса в релизную ветку

```bash
git switch release/1.x
git cherry-pick -x <hash-фикса-из-main>
git push
```

### Собрать несколько коммитов в один

```bash
git cherry-pick -n <c1> <c2> <c3>    # без коммитов, всё в индекс
git commit -m "Перенести подборку изменений X"
```

### Перенести «не туда» закоммиченное

```bash
git switch right-branch
git cherry-pick <hash>               # скопировать на нужную ветку
git switch wrong-branch
git reset --hard HEAD~1              # убрать из исходной (если не запушено)
```

## 🆚 cherry-pick против rebase, merge, revert

| Команда | Что делает |
| :--- | :--- |
| **`cherry-pick`** | копирует **выбранные** коммиты поверх текущей ветки |
| [**`rebase`**](rebase.md) | переносит **всю** серию коммитов ветки на новую базу |
| [**`merge`**](merge.md) | объединяет ветки целиком (с общей историей) |
| [**`revert`**](revert.md) | применяет **обратные** правки коммита (отмена) |

> [!warning] Дубликаты при последующем merge
> cherry-pick делает **копию** с новым хешем. Если потом слить исходную ветку, Git может не распознать, что коммит уже перенесён, → возможны дубли/конфликты. Для систематического переноса серий лучше [`rebase`](rebase.md); cherry-pick — для точечных, разовых задач.

## 🩺 Частые вопросы

- **«bad revision» / «empty commit»** — изменения уже есть в текущей ветке (коммит пустой после применения): `--skip` или `git cherry-pick --allow-empty`.
- **Откуда пришёл коммит** — используй `-x`, чтобы остался след `(cherry picked from …)`.
- **cherry-pick merge-коммита** — нужен `-m 1` (как в [revert](revert.md)).
- **Запутался в конфликте серии** — `git cherry-pick --abort` вернёт всё в исходное.
- **Не хочу сразу коммитить** — `-n`, потом один свой `git commit`.

## 💎 Резюме

- `git cherry-pick <commit>` — копия конкретного коммита поверх текущей ветки (новый хеш).
- Диапазон: `A..B` (без A) или `A^..B` (с A); несколько — перечислением.
- Бэкпорт со следом: `-x`; собрать в один коммит: `-n` + `git commit`.
- Конфликт: правишь → `add` → `--continue` (или `--skip`/`--abort`).
- Точечно — cherry-pick; серию на новую базу — [`rebase`](rebase.md).
