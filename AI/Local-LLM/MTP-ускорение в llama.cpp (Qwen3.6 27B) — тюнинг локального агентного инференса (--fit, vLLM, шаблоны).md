# ⚡ MTP-ускорение в llama.cpp (Qwen3.6 27B) — тюнинг локального агентного инференса

Пост-комментарий к треду с r/LocalLLaMA: команда гоняет **Qwen3.6 27B с MTP** в кванте **Q8_K_XL** через [llama.cpp](llama.cpp%20%E2%80%94%20%D0%B4%D0%B2%D0%B8%D0%B6%D0%BE%D0%BA%20%D0%BB%D0%BE%D0%BA%D0%B0%D0%BB%D1%8C%D0%BD%D0%BE%D0%B3%D0%BE%20%D0%B8%D0%BD%D1%84%D0%B5%D1%80%D0%B5%D0%BD%D1%81%D0%B0%20GGUF.md) на **RTX PRO 6000** (~55–60 tok/s, контекст 262K) и упирается в баги связки с **Copilot для VS Code**: случайные падения и пустые ответы из-за «битых» reasoning-блоков. Дальше — набор рабочих обходов (флаги шаблонов, `--fit`, миграция параллельных агентов на **vLLM**). Заметка разбирает, **что из этого правда** (на удивление — почти всё, это свежие фишки 2026-го) и как переложить на твои Linux-системы.

> [!note] Что проверено по первоисточникам — это не выдумка поста
> Я сперва заподозрил фантазию (странные «Qwen3.6», «Qwopus», флаг `--fit`), но всё подтвердилось:
> - **Qwen3.6-27B-MTP** реально существует ([unsloth/Qwen3.6-27B-MTP-GGUF](https://huggingface.co/unsloth/Qwen3.6-27B-MTP-GGUF)): MTP-головы для спекулятивного декодинга, нативный контекст **262 144** (до ~1M через RoPE), квант **UD-Q8_K_XL ≈ 35.8 ГБ**.
> - **MTP (Multi-Token Prediction)** влит в официальный llama.cpp (PR #22673, ~май 2026) — отдельный форк больше не нужен.
> - Флаг **`--fit`** в llama.cpp **реален** и включён по умолчанию: авто-раскладка модели по VRAM/RAM.
> - **Qwopus** — реальный файнтюн ([Jackrong/Qwopus3.6-27B](https://huggingface.co/Jackrong/Qwopus3.6-27B-v1-preview)), Qwen3.6-27B, дообученный на «Opus»-ризонинге (родственник [Qwythos-9B](../Model/Qwythos-9B-Claude-Mythos-5-1M%20%E2%80%94%20%D1%80%D0%B0%D1%81%D1%86%D0%B5%D0%BD%D0%B7%D1%83%D1%80%D0%B5%D0%BD%D0%BD%D1%8B%D0%B9%20%D1%84%D0%B0%D0%B9%D0%BD%D1%82%D1%8E%D0%BD%20Qwen3.5-9B%20%D0%BD%D0%B0%20%D1%82%D1%80%D0%B5%D0%B9%D1%81%D0%B0%D1%85%20Claude%20%281M%20%D0%BA%D0%BE%D0%BD%D1%82%D0%B5%D0%BA%D1%81%D1%82%29.md)).

## 🧩 Что такое MTP и почему он ускоряет

**MTP / Multi-Token Prediction** — у модели есть встроенные «черновые» головы, которые предсказывают **сразу несколько токенов вперёд**, а основная сеть их **проверяет одним проходом** (вид спекулятивного декодинга, но без отдельной draft-модели). Принятые токены — бесплатное ускорение, отклонённые — откат.

> [!warning] Про цифры ускорения: «15–20%» против «×1.5–2» — оба верны, это про разный режим
> - Карточка Unsloth и бенчи обещают **×1.5–2** (напр. 38→65 tok/s на RTX 3090). Это **одиночный поток, короткий контекст** — лучший случай.
> - Пост говорит **15–20% без потери качества** — это честная **реальная** цифра под **агентным** профилем: длинный контекст (262K) + параллельные запросы. При батчинге GPU и так загружен, и выигрыш спекулятивного декодинга **схлопывается**. Чем выше acceptance-rate черновых токенов и короче контекст — тем ближе к ×2.
> - Вывод: **не жди ×2 на своих агентских задачах**; 15–20% — нормальный честный прирост, а «×2» — маркетинговый максимум.

> [!caution] «55–60 tok/s» и Q8_K_XL — это на железе за $$$, не на обычной карте
> - **RTX PRO 6000** (Blackwell) — **96 ГБ VRAM**. Q8_K_XL весит **~36 ГБ** + KV-кэш на 262K контекста ⇒ нужна карта на **40 ГБ+**. На 24-гигабайтной (3090/4090) Q8 не влезет — бери меньший квант (Q4_K_M/Q5) и/или `--fit` с оффлоадом в RAM (медленнее).
> - **Сам тред на Reddit я подтвердить не смог** (Reddit блокирует фетч, curl → 403). Числа tok/s — анекдотические, из обсуждения; воспринимай как ориентир, проверяй `llama-bench`-ем на своём железе.

## 🛠️ Обходы из обсуждения (применимы к llama.cpp)

| Проблема | Решение | Флаги |
| :--- | :--- | :--- |
| Copilot/VS Code: пустые ответы, парсер не видит reasoning | задать **явный chat-template** | `--jinja --chat-template-file <шаблон>` |
| конфликт формата reasoning-блоков | **не навязывать** deepseek-формат рассуждений | убрать `--reasoning-format deepseek` |
| модель не влезает в VRAM / фрагментация | **авто-раскладка** памяти GPU↔CPU | `--fit on` (по умолч.), `--fit-ctx <N>`, `--fit-target <MiB>` |
| лишний расход памяти/скорости | **урезать контекст** до реальных нужд задачи | `-c 32768` вместо 262K |
| много **параллельных** агентов | для конкурентного сервинга llama.cpp на Windows — тупик | мигрировать на **vLLM** (Linux/WSL) |

> [!tip] Запуск MTP-кванта в llama.cpp
> ```bash
> # MTP-веса берём из MTP-репозитория (обычный GGUF голов не содержит)
> llama-server -hf unsloth/Qwen3.6-27B-MTP-GGUF:UD-Q4_K_XL \
>   --spec-type draft-mtp --spec-draft-n-max 2 \   # включить MTP-черновик
>   --jinja \                                       # явный chat-template (лечит Copilot)
>   --fit on --fit-ctx 32768 \                      # авто-память + гарантия контекста
>   -c 32768 --host 127.0.0.1 --port 8080
> ```
> Точные флаги MTP сверяй с карточкой модели — API спекулятивного декодинга в llama.cpp ещё меняется.

## 💻 Как это ложится на твои системы

Хорошая новость: пост сам гонит с Windows на **Linux** — это прямо твой случай.

| Система | Применимость |
| :--- | :--- |
| **Gentoo / Arch / Debian** (десктоп с NVIDIA) | **Основной путь.** llama.cpp + CUDA — ставится из исходников/AUR (см. [заметку про llama.cpp](llama.cpp%20%E2%80%94%20%D0%B4%D0%B2%D0%B8%D0%B6%D0%BE%D0%BA%20%D0%BB%D0%BE%D0%BA%D0%B0%D0%BB%D1%8C%D0%BD%D0%BE%D0%B3%D0%BE%20%D0%B8%D0%BD%D1%84%D0%B5%D1%80%D0%B5%D0%BD%D1%81%D0%B0%20GGUF.md)). 27B Q4 комфортно от **24 ГБ VRAM**, Q8 — от 40 ГБ. Для **многопоточного** сервинга агентов — **vLLM** (`pip install vllm`, нужен CUDA): на Linux планировщик и paged-attention заточены под параллель. |
| **Windows** | По выводу обсуждения — **тупик для параллельных агентов**: одиночный llama.cpp работает, но конкурентный сервинг лучше через **vLLM в WSL2/Linux**. |
| **Entware / RT-AX56U** (armv7, 512 МБ, без GPU) | **N/A.** 27B не запустить даже близко; MTP/спекулятивка тут не про роутер. Максимум — лёгкий прокси к десктопному `llama-server` (см. [llama.cpp](llama.cpp%20%E2%80%94%20%D0%B4%D0%B2%D0%B8%D0%B6%D0%BE%D0%BA%20%D0%BB%D0%BE%D0%BA%D0%B0%D0%BB%D1%8C%D0%BD%D0%BE%D0%B3%D0%BE%20%D0%B8%D0%BD%D1%84%D0%B5%D1%80%D0%B5%D0%BD%D1%81%D0%B0%20GGUF.md)). |

## 🔍 Прочее из поста

- **Qwopus** — файнтюн Qwen3.6-27B на «Opus»-ризонинге; в обсуждении хвалят за **стабильность агентных циклов**. Концептуально то же, что [Qwythos-9B](../Model/Qwythos-9B-Claude-Mythos-5-1M%20%E2%80%94%20%D1%80%D0%B0%D1%81%D1%86%D0%B5%D0%BD%D0%B7%D1%83%D1%80%D0%B5%D0%BD%D0%BD%D1%8B%D0%B9%20%D1%84%D0%B0%D0%B9%D0%BD%D1%82%D1%8E%D0%BD%20Qwen3.5-9B%20%D0%BD%D0%B0%20%D1%82%D1%80%D0%B5%D0%B9%D1%81%D0%B0%D1%85%20Claude%20%281M%20%D0%BA%D0%BE%D0%BD%D1%82%D0%B5%D0%BA%D1%81%D1%82%29.md) — Qwen, дообученный на трейсах Claude-семейства (бенчмарки самозаявленные, проверяй на своих задачах).
- **Context7 / Serper** — внешние инструменты ([MCP](../MCP%20%E2%80%94%20%D1%81%D0%B5%D1%80%D0%B2%D0%B5%D1%80%D1%8B%20Model%20Context%20Protocol.md)-сервер с актуальной докой / поисковый API). Реально полезны: локальная модель не знает свежих API и фактов — инструменты закрывают этот пробел.
- **Стоимость:** тезис «локальный стек заменит облако по деньгам» верен только при **уже купленном** дорогом GPU; иначе амортизация железа съедает экономию.

## 💡 Кому полезно

- **Запускаешь 27B-класс локально** на NVIDIA и хочешь выжать скорость — включай MTP, мерь `llama-bench`-ем, не верь маркетинговому «×2» вслепую.
- **Поднимаешь несколько агентов параллельно** — сразу смотри в сторону **vLLM на Linux**, а не llama.cpp.
- **Ловишь пустые ответы в VS Code/Copilot** с локальной моделью — лечится `--jinja` + явным шаблоном и снятием навязанного reasoning-формата.
- **Не** для роутера/слабого железа и **не** «бесплатно» без дорогой видеокарты.
