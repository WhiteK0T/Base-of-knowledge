# 🔁 Loops! — каталог workflow-петель для Claude Code и агентов

**Loops!** ([loops.elorm.xyz](https://loops.elorm.xyz/)) — community-каталог готовых **«замкнутых» рабочих сценариев (closed-loop workflows)** для кодинг-агентов. Идея: вместо того чтобы сто раз объяснять агенту одно и то же, берёшь готовую петлю, в которой уже прописаны **триггеры, контрольные точки (feedback gates) и условия выхода** — агент сам «крутит» задачу, пока не доведёт до конца.

## 🧬 Что такое «петля» (loop)

Closed-loop workflow — это сценарий, где агент **самостоятельно итерирует** до выполнения цели:

```text
триггер → действие → проверка (gate) → не готово? повторить → готово? выход
```

Каждая петля описывает: **когда** запускаться, **что** делать, **как** проверять результат и **когда** остановиться. За счёт этого агент «самопейсится» (self-pace) без ручного пинка на каждом шаге.

> [!info] Близко к идее /loop в Claude Code
> Это та же логика, что у повторяющихся самоуправляемых задач: «делай и проверяй, пока не достигнешь условия выхода». Loops! — это **готовые рецепты** таких циклов под типовые задачи.

## 🗂️ Категории и примеры

| Категория | Примеры петель |
| :--- | :--- |
| **CI** | `Ship PR Until Green`, `CI Failure Watcher` |
| **Testing** | `Post-Edit Test Guard`, `Pre-Commit Guard` |
| **Verification** | `Independent Verifier Pass` |
| **DevOps** | `Deploy Verification Loop` |

Каждая петля показана со счётчиками использования (напр. «979 копий · 1.9k установок») — видно, что реально популярно.

## 🚀 Как использовать

Для каждой петли — несколько способов подключения:

1. **Скопировать kickoff-инструкции** — готовый промпт-затравку вставить агенту.
2. **Открыть в IDE** — сразу в редактор.
3. **Установить hook-бандл** — автоматизация через хуки агента (для Claude Code это [хуки](Claude%20Code%20%E2%80%94%20гайд.md), срабатывающие на события).

**Совместимость:** Claude Code, Cursor, Codex, Gemini CLI, OpenCode.

**Свой вклад:** есть кнопка «Submit yours» — петли присылает сообщество.

## ⚠️ На что обратить внимание

> [!warning] Community-контент — проверяй перед запуском
> Петли присылают пользователи; качество и безопасность **разнятся**. Особенно осторожно с теми, что ставят **хуки** или сами запускают команды (тесты, деплой, git) — прочитай, что именно выполняет петля, прежде чем дать ей автономность. Не запускай вслепую на проде/важном репозитории.

> [!note] Бесплатно, но на донатах
> Сайт просит поддержки «чтобы проект оставался бесплатным и живым». Явной лицензии/гарантий аптайма нет — полезные петли имеет смысл **сохранять себе** (промпт + описание), чтобы не зависеть от доступности сайта.

## 💡 Кому полезно

- Часто гоняешь агента по **рутине** (прогнать тесты до зелёного, держать PR «green», проверять деплой) — готовые петли экономят формулировку.
- Хочешь подсмотреть **паттерны** надёжных автономных циклов (триггеры/гейты/выход) и адаптировать под себя.
- Для разовых уникальных задач проще написать инструкцию руками — петли выигрывают на **повторяемом**.
