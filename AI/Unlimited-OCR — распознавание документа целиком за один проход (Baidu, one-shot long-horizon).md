# 📄 Unlimited-OCR — распознавание документа целиком за один проход (Baidu, one-shot long-horizon)

[**Unlimited-OCR**](https://github.com/baidu/Unlimited-OCR) ([веса на HF](https://huggingface.co/baidu/Unlimited-OCR), [arXiv 2606.23050](https://arxiv.org/pdf/2606.23050)) — открытая **vision-language модель (VLM) для распознавания документов** от **Baidu**, развивающая идеи **DeepSeek-OCR**. Лозунг — *«One-shot long-horizon parsing»*: модель читает **весь документ целиком за один проход**, а не постранично/кусками. ~**3B** параметров (BF16), лицензия **MIT**, ~8.4k загрузок/мес. На вход — изображения (JPG/PNG) и **PDF**, на выход — структурированный **Markdown**.

## 🧩 Зачем читать документ целиком (а не постранично)

Классический пайплайн «OCR → LLM» **режет документ на куски** (страницы и мельче), распознаёт каждый отдельно и потом «сшивает» — потому что текст страницы это много токенов и целая книга не влезает в контекст. Отсюда боль:

- **Разрывы на границах** — таблица/абзац/формула через страницу рвутся пополам, половинки выглядят несвязанными.
- **Потеря порядка чтения** — многоколоночная вёрстка, сноски, врезки, колонтитулы путают поток текста.
- **Дубли и мусор** — повторяющиеся колонтитулы/номера страниц на каждом куске попадают в текст десятки раз.
- **Накладные расходы** — N проходов + логика склейки + перекрытия кусков ради контекста.

Когда модель **видит документ целиком**, она держит весь контекст сразу:

- **сквозная связность** (таблицы/абзацы/рисунки через границы собираются как единое целое);
- **правильный reading order** при сложной вёрстке;
- **единая структура на выходе** — одна сквозная иерархия заголовков, один Markdown/JSON, а не N кусков на склейку;
- **меньше ошибок и дублей** — нет «швов» от чанкинга;
- **скорость** — один inference-проход вместо «нарезать → много раз прогнать → склеить».

> [!info] Ключевая идея — vision-token compression (из DeepSeek-OCR)
> Страница скармливается модели **как изображение**, и кодируется в **десятки раз меньшим числом токенов**, чем тот же текст символами (картинка страницы ≈ сотни визуальных токенов против тысяч текстовых). Поэтому **целый документ умещается в окно контекста** и парсится за один проход. Отсюда и заявленное «ускорение в сотни раз» — следствие сжатия + отказа от многопроходной склейки.

## 🧪 Факты против хайпа

> [!warning] «Unlimited» — это бренд, не буквально бесконечность
> - В модели фигурирует **предел ~32 768 токенов** и **режим Base (image_size 1024)** именно для многостраничных PDF; режим **Gundam (640×1024)** — для одиночных картинок. Реально **огромную книгу (сотни страниц) за один присест может не хватить** — лимиты есть.
> - **«Скорее всего добавят себе OpenAI и Anthropic»** — спекуляция автора поста, без подтверждений.
> - **Это не магия с нуля:** авторы сами благодарят DeepSeek-OCR / DeepSeek-OCR-2 и PaddleOCR — это шаг развития, а не революция на пустом месте.
> - **«Сканируем тут» ≠ зашёл на сайт:** это VLM-модель, нужен Python/PyTorch + GPU (см. ниже), а не веб-кнопка.

## ⚙️ Запуск

Это VLM на ~3B: на вход картинки/PDF, запуск через `transformers` (или **vLLM** / **SGLang** / Docker Model Runner). **Нужна NVIDIA-GPU с CUDA** (CPU-инференс возможен, но мучительно медленный); для 3B в BF16 ориентируйся на **~8–16 ГБ VRAM**.

```python
import torch
from transformers import AutoModel, AutoTokenizer

model = AutoModel.from_pretrained(
    "baidu/Unlimited-OCR",
    trust_remote_code=True,
    torch_dtype=torch.bfloat16,
).eval().cuda()
# режим Base — для многостраничных PDF; Gundam — для одиночных изображений
# выход — Markdown со структурой документа
```

Версии из карточки: `torch==2.10.0`, `transformers==4.57.1`, **CUDA 12.9+**.

| Система владельца | Применимость |
| :--- | :--- |
| **Gentoo** (основная) | при наличии **NVIDIA-GPU**: venv + `pip install torch transformers`, CUDA-тулкит (`emerge dev-util/nvidia-cuda-toolkit`), драйвер `x11-drivers/nvidia-drivers`. Без GPU — только медленный CPU |
| **Debian / Ubuntu** | venv + `pip install torch transformers` с CUDA-колёсами, пакет `nvidia-cuda-toolkit` + проприетарный драйвер; либо контейнер с CUDA |
| **Arch** (план с июня 2026) | `python-pytorch-cuda` (или pip-venv), `cuda`, `nvidia` из репозиториев; vLLM/SGLang из AUR |
| **Entware / RT-AX56U** | **N/A** — 3B VLM требует CUDA-GPU и гигабайты VRAM; на роутере (armv7, 512 МБ RAM) невозможно |

> [!tip] Когда брать
> Нужно **массово и качественно** превращать **сканы/PDF/книги** в Markdown/текст для пайплайна (RAG, архив, датасет) — особенно длинные документы со сложной вёрсткой, таблицами и формулами, где постраничный OCR разваливается. Для разовой пары страниц проще онлайн-OCR или [PaddleOCR](https://github.com/PaddlePaddle/PaddleOCR).

## 💡 Кому полезно

- **RAG/датасеты:** аккуратная структурная оцифровка длинных PDF/книг (таблицы, формулы, порядок чтения) без «швов» чанкинга.
- **Архивы/оцифровка:** превратить сканы в чистый Markdown.
- **Не** для роутера/слабого железа без GPU и **не** «волшебная кнопка на сайте».
