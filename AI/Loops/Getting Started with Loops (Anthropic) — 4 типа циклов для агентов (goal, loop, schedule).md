# 🔁 Getting Started with Loops (Anthropic) — 4 типа циклов для агентов

Anthropic (команда Claude Code) выпустила официальное руководство **[«Getting started with loops»](https://claude.com/blog/getting-started-with-loops)** про **loop engineering** — подход «проектируй цикл, а не пиши разовый промпт». Идея: агент сам крутит цикл «работа → проверка → повтор» до достижения **условия остановки**. Гайд разбирает **4 типа циклов** (лестница: от ручного к автономному), как проверять результат, задавать условия выхода и не жечь токены. Это концептуальный компаньон к community-каталогу готовых рецептов [Loops!](Loops%20%E2%80%94%20%D0%BE%D0%B1%D0%B7%D0%BE%D1%80%20%D1%81%D0%B0%D0%B9%D1%82%D0%B0%20%D0%B8%20%D0%BA%D0%B0%D1%82%D0%B0%D0%BB%D0%BE%D0%B3%20%D0%BF%D0%B5%D1%82%D0%B5%D0%BB%D1%8C.md).

## 🪜 Четыре типа циклов

| # | Тип | Примитив Claude Code | Когда применять |
| :--- | :--- | :--- | :--- |
| 1 | **Агентный** (turn-based) | нет команды — режим по умолчанию | короткие разовые задачи, ты сам направляешь каждую итерацию |
| 2 | **Целевой** (goal-based) | `/goal` | есть проверяемый критерий «готово», одного хода мало |
| 3 | **Временной по расписанию** (time-based) | `/loop` (локально) · `/schedule` (в облаке) | повторяющиеся задачи и мониторинг систем, меняющихся по времени |
| 4 | **Полностью автономный** (proactive) | `/schedule` + `/goal` + skills + dynamic workflows + auto mode | поток однотипной работы без участия человека (триаж багов, зависимости) |

### 1️⃣ Агентный (turn-based)

«Агент повторяет циклы работы до условия остановки», но запуск **ручной** — ты промптишь, Claude сам оценивает готовность или просит контекст. Команды нет, это дефолтный режим общения.

> [!tip] Как усилить проверку
> Закодируй ручные проверки в **`SKILL.md`** и делай их **количественными**: *«чем более количественные проверки, тем легче Claude самопроверяться»*. Пример: не «посмотри, норм ли», а «подними dev-сервер → потыкай изменения → проверь консоль браузера на ошибки → прогони перф-аудит».

### 2️⃣ Целевой (goal-based) — `/goal`

Claude итерирует, пока не достигнет цели **или** не упрётся в лимит ходов. Каждый раз, когда он пытается остановиться, **модель-оценщик (evaluator) проверяет условие** и возвращает его в работу, пока цель не достигнута.

```text
/goal подними Lighthouse главной до 90+, остановись после 5 попыток
```

> [!note] Критерий должен быть детерминированным
> Задавай измеримое (число прошедших тестов, порог метрики), а не субъективное «сделай хорошо». **Ставь лимит ходов** — это прямой контроль стоимости.

### 3️⃣ Временной по расписанию (time-based) — `/loop` · `/schedule`

Цикл запускается **с заданным интервалом** для повторной работы или мониторинга внешних систем; крутится, пока не отменишь вручную или работа не завершится.

```text
/loop 5m проверь мой PR, ответь на комментарии ревью и почини упавший CI
```

- **`/loop`** — гоняет **локально**; **`/schedule`** — переносит рутину в **облако**.
- **Правило интервала:** частота = как часто реально меняется наблюдаемая система. *«Не запускай рутину чаще, чем нужно»* — иначе жжёшь токены впустую.

### 4️⃣ Полностью автономный (proactive)

Работа по событию/расписанию **без вмешательства человека в реальном времени**: отдельные задачи выходят по достижении цели, но сама рутина крутится, пока её не остановят. Собирается из нескольких примитивов сразу.

```text
/schedule каждый час: проверяй #project-feedback на баг-репорты.
/goal: не останавливайся, пока каждый найденный за прогон репорт
не затриажен, обработан и на него не дан ответ.
При фиксе бага — workflow: три решения параллельно в worktree,
и судья состязательно их ревьюит.
```

> [!tip] Экономия на масштабе
> Роутинг простых шагов на **меньшие/быстрые модели**, а «судейские» решения — на самую способную. Разные агенты для ревью, чтобы избежать предвзятости.

## ✅ Проверка результата и условия остановки

- **Чёткие границы.** У цикла должны быть явные критерии успеха и остановки — это помогает Claude прийти к решению **раньше (но не слишком рано)**.
- **Количественные чек-и** вместо субъективных: пороги, счётчики тестов, коды выхода.
- **Инструменты верификации** держи под рукой (скрипты, линтеры), а **ревью** отдавай **отдельному агенту** — так меньше предвзятости.
- Для `/goal` работу принимает **модель-оценщик**; всегда задавай **max ходов** как страховку.

## 🔥 Как не жечь токены

- **Модель и effort-уровень — самый большой рычаг стоимости** цикла. Выбирай их под сложность задачи.
- **Подбирай примитив под задачу** — не гоняй автономный proactive там, где хватит одного `/goal`.
- **Скрипты для детерминированной работы** вместо «рассуждений» модели.
- **Обкатывай dynamic workflows на малых данных** прежде чем масштабировать.
- **Смотри расход**: разбивка по `/usage` и счётчики токенов у `/goal`.
