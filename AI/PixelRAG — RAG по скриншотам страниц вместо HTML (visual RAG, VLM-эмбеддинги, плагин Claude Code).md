# 🖼️ PixelRAG — RAG по скриншотам страниц вместо HTML (visual RAG, VLM-эмбеддинги)

[**PixelRAG**](https://github.com/StarTrail-org/PixelRAG) (StarTrail-org, **Python**, Apache-2.0, ~5200★) — система **визуального RAG**: документы (веб-страницы, PDF, картинки) **рендерятся в скриншоты**, и поиск/извлечение идёт **прямо по изображениям**, а не по распарсенному в текст HTML. Слоган проекта: *«The end of web parsing. The beginning of scalable pixel-native search»* — то есть это сознательно **не парсер**, а его противоположность: ничего не вытаскивается в текст, страница остаётся «как картинка», и за неё отвечает **vision-language модель**.

Идея: обычный текстовый RAG выкидывает визуальную структуру — таблицы, графики, вёрстку, расположение элементов. Если же эмбеддить **скриншот**, то схемы, таблицы и layout сохраняются, и на вопросы по ним можно отвечать напрямую из пикселей.

## ⚙️ Как это устроено

Конвейер: **render → chunk (тайлы) → embed → index → serve**.

| Этап | Компонент | Что делает |
| :--- | :--- | :--- |
| **Рендер** | `pixelshot` | гонит страницу/PDF через **Playwright/Chromium** (Chrome DevTools Protocol) и режет на **скриншот-тайлы** |
| **Эмбеддинг** | **Qwen3-VL-Embedding-2B** (LoRA-дообучена на скриншотах) | кодирует тайлы в векторы |
| **Индекс** | **FAISS** | векторный индекс для поиска |
| **Выдача** | **FastAPI** | эндпоинт `/search` (CPU или GPU) |

То есть «мозг» здесь — отдельная **VLM-эмбеддинг-модель Qwen3-VL**, а не сам Claude. Claude (или другой агент) — лишь **потребитель** результата: получает релевантные тайлы/ответ и работает с ними.

## 🚀 Как ставится и запускается

Вариантов несколько — от лёгкого плагина до полного self-hosted конвейера:

```bash
# 1) Плагин Claude Code (самый простой путь — «пиксельный браузер»)
claude plugin install pixelbrowse@pixelrag-plugins
# затем:
claude -p "screenshot https://news.ycombinator.com and summarize top stories"
```

```bash
# 2) Только рендер страницы в тайлы (ядро, без эмбеддингов)
pip install pixelrag
pixelshot https://en.wikipedia.org/wiki/Python --output ./tiles
```

```bash
# 3) Полный конвейер: индексировать свой PDF/сайт и поднять поиск локально
pip install 'pixelrag[index]'
# создать pixelrag.yaml с путём к источнику, затем:
pixelrag index build
pixelrag serve --index-dir ./my_index --port 30001
```

```bash
# 4) Готовый облачный индекс (8.28M страниц Wikipedia, без API-ключа)
curl -X POST https://api.pixelrag.ai/search \
  -H "Content-Type: application/json" \
  -d '{"queries": [{"text": "What is the capital of France?"}], "n_docs": 5}'
```

Экстры пакета: `pixelrag[embed]` (эмбеддинг+индексация), `pixelrag[serve]` (search API), `pixelrag[index]` (полная оркестрация).

> [!note] Что из этого «работает локально»
> - **Плагин `pixelbrowse`** для Claude Code действительно лёгкий: он вызывает **`pixelshot` локально** (делает скриншот, отдаёт картинку модели), **без бэкенда и MCP-сервера**. Вот это и есть «в пару кликов».
> - **Полноценный поиск по своему корпусу** (вариант 3) — уже не «пара кликов»: нужна **VLM-модель Qwen3-VL-Embedding-2B**, FAISS-индекс и желательно **GPU** (CUDA на Linux / MPS на macOS; на CPU работает, но медленно).
> - **Готовый индекс Wikipedia** (вариант 4) — это **облако** (`api.pixelrag.ai`), а не локальная установка. Сам пиксельный индекс Wikipedia ~**217 ГБ**.

## 🧪 Факты против хайпа

Пост («Claude научили полноценно читать сайты — вышел парсер нового уровня… устанавливается плагином в пару кликов и работает локально») передаёт суть, но смещает акценты:

> [!warning] Что в посте натянуто
> - **«Парсер нового уровня»** — наоборот. PixelRAG позиционируется как **конец парсинга**: он **намеренно не разбирает страницу в текст**, а заменяет это эмбеддингом скриншота. Называть его «парсером» — терминологически мимо.
> - **«Claude научили читать сайты»** — учат **не Claude**. Это **самостоятельный фреймворк** (Apache-2.0, Python), а «зрение» обеспечивает **отдельная модель Qwen3-VL-Embedding**. Claude — опциональный клиент через плагин, никаких изменений в самой модели нет.
> - **«В пару кликов и работает локально»** — верно **только для плагина-скриншотера**. Поднять собственный пиксельный поиск локально — это VLM + FAISS + (желательно) GPU, а «готовый» режим — облачный API.

> [!tip] Где подход реально силён
> Скриншот-RAG честно выигрывает там, где **текст теряет смысл**: сложные **таблицы, графики, диаграммы, инфографика, формы, многоколоночная вёрстка, отсканированные PDF**. Тут пиксели сохраняют то, что HTML/`pdftotext` рушат. Перекликается с [Unlimited-OCR](Unlimited-OCR%20%E2%80%94%20%D1%80%D0%B0%D1%81%D0%BF%D0%BE%D0%B7%D0%BD%D0%B0%D0%B2%D0%B0%D0%BD%D0%B8%D0%B5%20%D0%B4%D0%BE%D0%BA%D1%83%D0%BC%D0%B5%D0%BD%D1%82%D0%B0%20%D1%86%D0%B5%D0%BB%D0%B8%D0%BA%D0%BE%D0%BC%20%D0%B7%D0%B0%20%D0%BE%D0%B4%D0%B8%D0%BD%20%D0%BF%D1%80%D0%BE%D1%85%D0%BE%D0%B4%20%28Baidu%2C%20one-shot%20long-horizon%29.md) — там тоже ставка на «понять документ как изображение».

> [!caution] Цена пиксельного подхода
> - **Скриншоты дороже текста по токенам/латентности** для итоговой VLM — картинка тяжелее строки.
> - Для своего корпуса нужен **GPU** под эмбеддинг (на CPU — медленно), Chrome/Chromium под рендер (`CHROME_PATH` если не найдётся автоматически), для PDF — Poppler.
> - **Динамика страницы** (то, что не попало в видимый скриншот: скрытые блоки, состояние после кликов) в пиксель не попадёт — рендерится то, что видно.

## 💻 Как это ложится на твои системы

Это **десктопный/серверный Python-инструмент**, на роутер не для него:

| Система | Применимость |
| :--- | :--- |
| **Gentoo** (основная) | OK: Python 3.10+, Chromium (`emerge www-client/chromium` или `chromium-bin`), Playwright; для эмбеддинга — GPU/ROCm/CUDA. Плагин Claude Code — без проблем |
| **Debian / Ubuntu** | OK: `apt install chromium` (или Playwright-сборка), `pip install 'pixelrag[...]'`; GPU желателен для своего индекса |
| **Arch** (план с июня 2026) | OK: `pacman -S chromium python`, PyTorch/CUDA из репозиториев/AUR |
| **Entware / RT-AX56U** | **N/A.** Нужны Chromium-рендер + VLM-эмбеддинг (torch) — это десктоп/GPU-класс, на armv7 с 512 МБ ОЗУ не запускается. Максимум — **дёргать облачный `api.pixelrag.ai`** обычным `curl` с роутера |

> [!tip] Практический сценарий
> Если просто хочется, чтобы **агент видел страницу глазами** (а не ломал голову над HTML) — поставь **плагин `pixelbrowse`** в [Claude Code](Claude%20Code%20%E2%80%94%20%D0%B3%D0%B0%D0%B9%D0%B4.md) и проси «screenshot URL and …». Полноценный пиксельный поиск по **своей** базе документов поднимай только при наличии GPU и реальной потребности в таблично-графических корпусах.

## 💡 Кому полезно

- **RAG-инженерам/исследователям:** альтернатива text-chunking там, где визуальная структура критична (отчёты, дашборды, научные PDF с графиками).
- **Пользователям Claude Code:** лёгкий плагин «дать агенту скриншот страницы» вместо борьбы с DOM.
- **Не** замена обычному текстовому RAG для чистого текста (там он дороже и избыточен) и **не** «апгрейд самой модели» — это внешний слой извлечения.
