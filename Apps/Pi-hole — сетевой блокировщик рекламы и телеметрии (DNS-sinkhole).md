# 🕳️ Pi-hole — сетевой блокировщик рекламы и телеметрии (DNS-sinkhole)

**Pi-hole** ([pi-hole.net](https://pi-hole.net)) — «чёрная дыра для рекламы»: лёгкий локальный **DNS-сервер**, который блокирует рекламу, трекеры и телеметрию **на уровне сети, для всех устройств сразу** (ПК, телефоны, ТВ, IoT). Поднимается на любом всегда-включённом устройстве (Raspberry Pi, домашний сервер, NAS, контейнер), а домашняя сеть ходит за DNS через него. Реклама не докачивается, потому что домен просто не резолвится.

## 🧬 Как работает (DNS-sinkhole)

```text
устройство → DNS-запрос «ads.example.com» → Pi-hole
   ├─ домен в блок-листе?  → ответ 0.0.0.0 (никуда) → реклама не загружается
   └─ домен обычный?       → форвардит на upstream (1.1.1.1/9.9.9.9) → нормальный ответ
```

Pi-hole — это **DNS-фильтр**: он не «вырезает» рекламу со страницы, а не даёт устройству узнать адрес рекламного/трекерного сервера. Один Pi-hole закрывает всю сеть без настройки на каждом устройстве.

> [!info] Компоненты (Pi-hole v6)
> `pihole-FTL` — ядро: DNS-резолвер (на базе dnsmasq) + опциональный DHCP + **встроенный веб-сервер и REST API**. `gravity` — агрегатор блок-листов (собирает домены в одну базу). Веб-интерфейс — статистика и управление. В **v6 (2025)** выкинули lighttpd/PHP: веб и API теперь прямо в FTL, конфиг — `pihole.toml`.

## ✨ Что даёт

- Блокировка рекламы/телеметрии **для всей сети** одной точкой.
- **Блок-листы** (gravity) с автообновлением по расписанию; белые/чёрные списки, regex-правила.
- **Группы клиентов** — разные правила для разных устройств.
- Опциональный **DHCP-сервер**, conditional forwarding, **DNSSEC**.
- Подробный лог запросов и красивая статистика (что и куда «звонит домой»).

## 📦 Установка

### Вариант 1. Docker (рекомендую, кроссплатформенно)

Способ из новости. Работает везде, где есть Docker (Gentoo, Debian/Ubuntu, Arch). Пример `docker-compose.yml`:

```yaml
services:
  pihole:
    container_name: pihole
    image: pihole/pihole:latest
    ports:
      - "53:53/tcp"
      - "53:53/udp"
      - "80:80/tcp"            # веб-интерфейс /admin
      # - "67:67/udp"          # только если Pi-hole = DHCP-сервер
    environment:
      TZ: 'Europe/Moscow'
      FTLCONF_webserver_api_password: 'задай_свой_пароль'
      FTLCONF_dns_listeningMode: 'all'   # для bridge-сети Docker
    volumes:
      - './etc-pihole:/etc/pihole'       # сохранение базы/настроек
    cap_add:
      - SYS_TIME                         # для NTP (опц.)
      - SYS_NICE                         # приоритет процесса (опц.)
    restart: unless-stopped
```

```bash
docker compose up -d        # поднять
# веб-панель: http://<ip-хоста>/admin
```

> [!warning] v6: переименовали переменную пароля
> В Pi-hole v6 пароль задаётся через `FTLCONF_webserver_api_password` (старое `WEBPASSWORD` из v5 устарело). Не используй `--privileged` (с 2022.04+) — добавляй точечные `cap_add`. Том `/etc/dnsmasq.d` теперь нужен только при миграции с v5.

### Вариант 2. Нативный установщик (Debian/Ubuntu)

```bash
curl -sSL https://install.pi-hole.net | bash
```

Официальный скрипт поддерживает Debian/Ubuntu (и Fedora/CentOS). Ставит сервис напрямую (systemd), без контейнера.

### По платформам владельца

| Система | Как ставить |
| :--- | :--- |
| **Debian / Ubuntu** | нативный установщик `curl \| bash` **или** Docker |
| **Gentoo** | официального установщика нет → **Docker** (`emerge app-containers/docker`) |
| **Arch** | официально не поддерживается → **Docker** (надёжнее, чем AUR-сборки) |
| **Entware / RT-AX56U** | сам Pi-hole **не ставить** (тяжёлый стек, 256 МБ flash) — см. ниже |

> [!danger] На роутер целиком Pi-hole не ставят
> ASUS RT-AX56U (Entware, 256 МБ flash) не потянет полноценный стек Pi-hole (веб, БД запросов, gravity). Правильные варианты: **(а)** держать Pi-hole на всегда-включённом десктопе/Raspberry Pi/сервере, а на роутере в DHCP указать его как DNS для всей сети; **(б)** если хочется именно на роутере — смотреть в сторону более лёгкого **AdGuard Home** (есть armv7-сборки, ляжет на подключённый к роутеру USB-диск). Это другой инструмент, но та же идея «DNS-блокировка для всей сети».

## 🔌 Подключение всей сети (главный кейс)

Чтобы фильтровались **все** устройства разом — пропиши Pi-hole как DNS **на роутере**: в DHCP-настройках RT-AX56U укажи IP машины с Pi-hole как единственный DNS-сервер для LAN. Тогда любое устройство в сети автоматически ходит через фильтр, ничего на клиентах настраивать не нужно.

> [!tip] Дай Pi-hole статический IP
> У хоста с Pi-hole должен быть фиксированный адрес (статика или DHCP-резервация на роутере) — иначе при смене IP вся сеть останется без DNS. И задай резервный/upstream DNS осмысленно: при падении Pi-hole сеть не должна полностью терять интернет.

## 🧰 Полезные команды (CLI)

| Команда | Действие |
| :--- | :--- |
| `pihole -up` | обновить Pi-hole |
| `pihole -g` | обновить базу блок-листов (gravity) |
| `pihole -a -p` | сменить пароль веб-панели |
| `pihole disable 5m` | временно выключить блокировку на 5 минут |
| `pihole -t` | живой лог DNS-запросов (tail) |
| `pihole -q example.com` | проверить, в каких списках домен |

(В Docker — те же команды через `docker exec pihole pihole ...`.)

## 🧩 Усиление приватности (опционально)

- **unbound** рядом с Pi-hole → собственный **рекурсивный** резолвер, не зависишь от публичных upstream (1.1.1.1/8.8.8.8).
- **cloudflared / DoH** → шифрованный DNS до upstream (DNS-over-HTTPS), чтобы провайдер не видел запросы.

## ⚠️ Чего Pi-hole НЕ умеет

> [!warning] Не панацея
> Блокировка по DNS не убирает рекламу, которую отдают **с того же домена**, что и контент (часть YouTube/Facebook/Instagram-рекламы, многие in-app баннеры) — домен нельзя заблокировать, не сломав сервис. Также Pi-hole — **единая точка отказа** для DNS: упал/обновляется — сеть без резолвинга. Качество зависит от **актуальности блок-листов** (не добавляй сомнительные «мега-листы» вслепую — поломают сайты).

## 💡 Кому полезно

- Хочешь **одним устройством** очистить от рекламы/телеметрии всю домашнюю сеть, включая ТВ и IoT, которые иначе не настроить.
- Есть всегда-включённый сервер/Raspberry Pi/NAS под контейнер.
- Нужна **видимость**, кто и куда «звонит домой» (статистика запросов).
